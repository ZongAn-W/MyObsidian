# 第九章 关系数据库存储管理：重要知识点总结

> 来源：《数据库系统概论（第6版）》第9章。重点掌握数据库逻辑组织与物理组织、记录表示、块组织、关系表组织方式、索引结构、B+ 树索引、哈希索引和位图索引。

## 1. 本章学习主线

存储管理是 DBMS 的重要职责之一，直接影响查询优化和数据库物理设计。

本章主要回答：

1. 数据库中的表、记录、块在磁盘上如何组织？
2. 如何用索引减少 I/O，提高查询效率？
3. B+ 树、哈希索引、位图索引各适合什么场景？

核心主线：

```text
逻辑结构：表空间、段、分区、数据块
    -> 物理结构：文件、物理块、记录
    -> 记录组织：定长、变长
    -> 表组织：堆、顺序、多表聚簇、B+树、哈希
    -> 索引结构：顺序索引、辅助索引、B+树、哈希、位图
```

## 2. 数据组织概述

数据库是大量数据的有结构集合，数据组织的目标有两个：

- 存储效率高，节省存储空间。
- 存取效率高，访问速度快，I/O 代价小。

数据库的存储结构和存取路径对用户通常是隐藏的。用户使用 SQL 描述需要的数据，DBMS 负责把逻辑请求转换为物理访问。

## 3. 逻辑组织方式与物理组织方式

### 3.1 两种存储管理方式

常见方式：

| 方式 | 含义 | 例子 |
| --- | --- | --- |
| 每个数据库对象对应一个操作系统文件 | DBMS 借助操作系统文件管理数据 | 一些系统把表、索引映射到文件 |
| 整个数据库对应一个或若干文件 | DBMS 自己管理数据库空间 | Oracle、SQL Server 等常用 |

### 3.2 逻辑结构与物理结构的对应

数据库逻辑组织常见层次：

```text
表空间 -> 段 -> 分区 -> 数据块
```

物理组织常见层次：

```text
文件 -> 物理块 -> 记录
```

对应关系：

| 逻辑概念 | 物理概念 |
| --- | --- |
| 表空间 | 文件 |
| 数据块 | 物理块 |
| 记录 | 存储在物理块中的记录 |

要点：

- 物理块是存储分配和 I/O 的基本单位。
- 一个物理块可以存放多个记录。
- 一个表通常占用多个块。

## 4. 记录表示

关系表的记录可以采用定长记录和变长记录两种形式存储。

### 4.1 定长记录存储

定长记录方式中，每条记录占用相同大小的空间。

特点：

| 优点 | 缺点 |
| --- | --- |
| 容易定位记录和属性位置 | 对变长字段可能浪费空间 |
| 插入、删除、修改处理较简单 | 存储空间利用率可能较低 |

即使存在 `VARCHAR` 等变长字段，也按最大长度预留空间。

适合：

- 属性长度固定。
- 记录格式稳定。
- 需要快速定位记录。

### 4.2 变长记录存储

变长记录方式中，每条记录可以占用不同大小空间。

常见实现方式：

1. 在每个变长字段前记录字段长度。
2. 先存放定长字段，再用指针指向变长字段。
3. 定长字段与变长字段分开存放在不同块中。

特点：

| 优点 | 缺点 |
| --- | --- |
| 节省空间 | 记录定位和修改较复杂 |
| 适合字符串、BLOB 等变长数据 | 插入、删除可能引起记录移动 |

## 5. 块的组织

记录最终存放在物理块中。块内通常包括：

- 块头：记录块号、时间戳、空闲空间指针等元数据。
- 记录区：存放具体记录。
- 空闲空间：供插入或更新记录使用。

### 5.1 定长记录的块组织

定长记录可按顺序放入块中。

特点：

- 记录可在原位置直接修改。
- 删除后可把空间加入空闲链表。
- 插入同长度记录时可复用空闲位置。

### 5.2 变长记录的块组织

变长记录通常需要在块头维护偏移量表或指针。

常见处理：

- 记录从块尾部开始连续存放。
- 块头保存记录偏移量。
- 删除记录时修改偏移表并回收空间。
- 修改记录变长后可能需要移动记录。

## 6. 关系表的组织方式

常见关系表组织方式：

| 方式 | 含义 |
| --- | --- |
| 堆存储 | 记录无序存放 |
| 顺序存储 | 记录按某个属性值有序存放 |
| 多表聚簇存储 | 相关表记录聚集存放 |
| B+ 树存储 | 记录按 B+ 树索引组织 |
| 哈希存储 | 记录按哈希函数分桶存放 |

### 6.1 堆存储

堆存储中，记录可放在表的任意块中，没有顺序要求。

优点：

- 插入简单。
- 只需找到有空闲空间的块。

缺点：

- 查询时通常需要全表扫描。
- 对范围查询、排序查询不友好。

### 6.2 顺序存储

顺序存储按指定属性或属性组的取值顺序存放记录。

优点：

- 适合按排序属性查找和范围查询。
- 可提高有序扫描效率。

缺点：

- 插入和更新可能需要移动记录。
- 维护记录顺序代价较高。

### 6.3 多表聚簇存储

多表聚簇存储把多个经常连接访问的表的相关记录存放在一起。

优点：

- 可减少连接操作的 I/O。
- 适合经常按外码连接的表。

缺点：

- 更新和插入可能更复杂。
- 对不按聚簇条件访问的查询可能不利。
- 可能出现频繁记录迁移。

### 6.4 B+ 树存储

B+ 树存储使用 B+ 树索引组织记录。

特点：

- 保持较高查询效率。
- 适合随机查找和范围查找。
- 插入、删除需要维护树结构。

### 6.5 哈希存储

哈希存储通过哈希函数计算属性值对应的桶号，把记录放到对应哈希桶中。

特点：

- 适合等值查询。
- 不适合范围查询和排序查询。
- 可能发生桶溢出。

## 7. 索引结构概述

索引类似书的目录，用额外的数据结构帮助快速定位记录。

索引的优点：

- 减少全表扫描。
- 可使用高效查找算法。
- 索引块可长期缓存在内存中，减少 I/O。

索引的代价：

- 占用额外存储空间。
- 插入、删除、更新记录时需要维护索引。
- 索引过多会降低更新效率。

常见索引：

- 顺序表索引。
- 辅助索引。
- B+ 树索引。
- 哈希索引。
- 位图索引。

## 8. 顺序表索引

顺序表索引建立在按索引属性值顺序存储的关系表上。

也称：

```text
主索引 / 聚簇索引
```

注意：这里“主索引”不等于“主码索引”。索引属性可以是任意属性组。

### 8.1 稠密索引

稠密索引中，基本表的每条记录都对应一个索引项。

索引项通常包含：

```text
索引属性值 + 指向相应记录的指针
```

优点：

- 查找精确。
- 可直接定位到记录。

缺点：

- 索引项数量多，索引文件较大。
- 维护代价较高。

### 8.2 稀疏索引

稀疏索引中，不是每条记录都有索引项，而是每个物理块或一组记录对应一个索引项。

索引项通常指向：

```text
某个存储块中的第一条记录
```

优点：

- 索引项较少，索引文件小。
- 插入、删除维护开销小。

缺点：

- 查找时定位到块后，还需在块内顺序查找。

### 8.3 多级索引

当一级索引仍然很大时，可以在索引上再建立索引，形成多级索引。

查找过程：

```text
高级索引 -> 低级索引 -> 数据块 -> 记录
```

多级索引可减少查找索引本身的 I/O。

## 9. 辅助索引

辅助索引是建立在表的非排序属性上的索引。

特点：

- 一个表最多只能有一个主索引。
- 可以在不同属性上建立多个辅助索引。
- 辅助索引通常必须是稠密索引。

原因：

```text
表不是按辅助索引属性有序存放的，
相同属性值的记录可能分散在很多块中。
```

若辅助索引属性重复值很多，索引项可能很大。可引入“指针桶”减少重复索引项。

## 10. B+ 树索引

### 10.1 B+ 树索引的特点

B+ 树是主流 DBMS 中最常用的索引结构之一。

特点：

- 平衡树，所有叶结点在同一层。
- 根到任一叶结点的路径长度相同。
- 非叶结点只存储键值和指针。
- 叶结点存储键值和指向记录的指针。
- 叶结点之间通常按键值顺序链接。

适合：

- 等值查询。
- 范围查询。
- 排序扫描。
- 前缀匹配等有序访问。

### 10.2 B+ 树查询

随机查找：

```text
从根结点开始
    -> 按键值比较选择子树
    -> 逐层向下
    -> 到叶结点
    -> 找到记录指针
```

范围查询：

```text
先找到范围起点所在叶结点
再沿叶结点链表顺序扫描
直到范围终点
```

### 10.3 B+ 树维护

插入：

- 找到应插入的叶结点。
- 若叶结点有空间，直接插入。
- 若叶结点溢出，分裂叶结点。
- 分裂可能向父结点传播，甚至导致根分裂，树高增加。

删除：

- 删除对应索引项。
- 若结点仍满足最小填充要求，结束。
- 若结点下溢，可能与兄弟结点合并或重分配。
- 合并可能向父结点传播，树高可能降低。

## 11. 哈希索引

### 11.1 哈希索引的结构

哈希索引由两个关键部分组成：

| 部分 | 作用 |
| --- | --- |
| 哈希函数 | 把索引属性值映射到桶号 |
| 哈希表/桶 | 存放索引项或记录 |

索引项通常包含：

```text
索引属性值 + 指向相应记录的指针
```

### 11.2 哈希函数

好的哈希函数应使记录尽量均匀地分布到各桶中。

如果分布不均，可能造成某些桶过满，影响查询效率。

### 11.3 桶溢出

桶空间有限，当桶放不下新索引项时发生桶溢出。

常见原因：

- 记录数超过预期。
- 属性取值分布不均。
- 哈希函数设计不好。

常见处理：

- 预留溢出空间。
- 使用溢出桶。
- 用溢出链把桶和溢出桶连接起来。

### 11.4 静态哈希与动态哈希

静态哈希：

- 桶数预先确定。
- 结构简单。
- 数据增长后容易发生溢出。

动态哈希：

- 随数据增长动态增加桶或目录。
- 更适合数据规模变化。
- 维护更复杂。

### 11.5 哈希索引适用场景

适合：

- 等值查询。
- 主码或唯一值查找。

不适合：

- 范围查询。
- 排序查询。
- `ORDER BY`、`GROUP BY` 中需要有序访问的场景。

## 12. 位图索引

### 12.1 位图索引的含义

位图索引是一组位向量。对于某个属性的每个可能取值，用一个位向量表示哪些记录具有该取值。

例子：属性 `Ssex` 只有“男”“女”两个取值，可建立两个位向量。

```text
男：1 表示该记录是男，0 表示不是
女：1 表示该记录是女，0 表示不是
```

### 12.2 位图索引的特点

适合：

- 基数小的属性，即取值种类少。
- 多条件组合查询。
- 统计查询。
- 读多写少的场景。

不适合：

- 基数大的属性，位图数量多，占用空间大。
- 更新频繁的表，维护成本高。

### 12.3 编码位图索引

编码位图索引用二进制编码表示属性取值，减少位向量个数。

若属性取值个数为 `K`：

- 普通位图索引需要 `K` 个位向量。
- 编码位图索引大约需要 `log2(K)` 个位向量。

代价：

- 查询时需要组合多个编码位向量。
- 逻辑判断可能更复杂。

## 13. 易混概念辨析

### 13.1 堆存储、顺序存储、哈希存储

| 方式 | 是否有序 | 适合场景 |
| --- | --- | --- |
| 堆存储 | 无序 | 插入频繁、简单存储 |
| 顺序存储 | 有序 | 范围查询、排序访问 |
| 哈希存储 | 按哈希桶 | 等值查询 |

### 13.2 稠密索引与稀疏索引

| 类型 | 索引项数量 | 查找 |
| --- | --- | --- |
| 稠密索引 | 每条记录一个索引项 | 可直接定位记录 |
| 稀疏索引 | 每块或每组记录一个索引项 | 先定位块，再块内查找 |

### 13.3 主索引与辅助索引

| 类型 | 建立位置 | 数量 |
| --- | --- | --- |
| 主索引/聚簇索引 | 建在顺序存储属性上 | 通常一个 |
| 辅助索引 | 建在非排序属性上 | 可多个 |

### 13.4 B+ 树索引与哈希索引

| 索引 | 优点 | 不适合 |
| --- | --- | --- |
| B+ 树 | 等值、范围、排序都较好 | 维护比简单哈希复杂 |
| 哈希 | 等值查询很快 | 范围查询、排序查询 |

### 13.5 位图索引与 B+ 树索引

| 索引 | 适合属性 |
| --- | --- |
| 位图索引 | 低基数属性，如性别、状态 |
| B+ 树索引 | 高基数属性、范围查询属性 |

## 15. 速记版

```text
存储管理目标：
省空间 + 少 I/O

记录：
定长好定位，变长省空间

表组织：
堆无序，顺序有序，聚簇少连接，B+树适合范围，哈希适合等值

索引作用：
像目录，减少全表扫描

稠密索引：
每条记录一个索引项

稀疏索引：
每块或每组一个索引项

B+树：
平衡、有序、叶子相连，支持等值和范围

哈希：
哈希函数 + 桶，适合等值，不适合范围

位图：
低基数属性，多条件查询用位运算快
```
