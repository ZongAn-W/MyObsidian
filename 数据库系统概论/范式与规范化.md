范式是衡量关系模式规范化程度的标准。

常见范式层次：

```text
4NF ⊂ BCNF ⊂ 3NF ⊂ 2NF ⊂ 1NF
```

也可以理解为：

```text
1NF -> 2NF -> 3NF -> BCNF -> 4NF
规范化程度逐步提高
异常和冗余逐步减少
```

### 6.1 1NF

若关系模式中每个属性都是不可再分的数据项，则属于第一范式 `1NF`。

1NF 是关系数据库中关系模式的最低要求。

### 6.2 2NF

定义：

若 `R ∈ 1NF`，且每一个非主属性都完全函数依赖于任何一个候选码，则 `R ∈ 2NF`。

通俗判断：

```text
2NF 消除非主属性对码的部分函数依赖
```

常见问题场景：

- 候选码是组合属性。
- 某些非主属性只依赖组合码的一部分。

例子：

```text
(Sno, Cno) -> Grade
Sno -> School
Sno -> Sloc
```

如果关系为 `SLC(Sno, School, Sloc, Cno, Grade)`，候选码是 `(Sno, Cno)`，但 `School`、`Sloc` 只依赖 `Sno`，因此不满足 2NF。

分解思路：

```text
SC(Sno, Cno, Grade)
S-L(Sno, School, Sloc)
```

### 6.3 3NF

定义要点：

若 `R ∈ 1NF`，并且不存在非主属性对码的传递函数依赖，则 `R ∈ 3NF`。

通俗判断：

```text
3NF 消除非主属性对码的部分依赖和传递依赖
```

例子：

```text
Sno -> School
School -> Sloc
所以 Sno -T-> Sloc
```

若 `Sloc` 是非主属性，则关系不满足 3NF。

分解思路：

```text
S-SS(Sno, School)
S-SL(School, Sloc)
```

### 6.4 BCNF

BCNF 是 Boyce-Codd 范式，通常看作修正的第三范式，比 3NF 更严格。

定义：

若 `R ∈ 1NF`，对于每一个非平凡函数依赖：

```text
X -> Y
```

都有 `X` 含有码，则 `R ∈ BCNF`。

通俗判断：

```text
BCNF 要求每一个决定因素都包含码。
```

BCNF 的结论：

- 若 `R ∈ BCNF`，则 `R ∈ 3NF`。
- 若 `R ∈ 3NF`，不一定 `R ∈ BCNF`。
- BCNF 消除了主属性和非主属性对码的部分依赖、传递依赖。

易错点：

3NF 允许某些“右部为主属性”的依赖存在；BCNF 不看右部是不是主属性，只看左部是不是超码。

### 6.5 多值依赖

多值依赖用于描述一个属性值对应另一组属性值的集合，而且这个集合与其他属性无关。

记作：

```text
X ->-> Y
```

含义：

在关系 `R(U)` 中，若给定 `X` 的值后，`Y` 的一组值只由 `X` 决定，与 `Z = U - X - Y` 的取值无关，则 `Y` 多值依赖于 `X`。

典型例子：

```text
Teaching(C, T, B)
```

一门课程 `C` 有多名教师 `T`，同时有多本参考书 `B`。若教师集合和参考书集合彼此独立，则：

```text
C ->-> T
C ->-> B
```

若放在一个关系中，会产生教师和参考书的笛卡尔组合，导致大量冗余。

### 6.6 4NF

定义：

若 `R ∈ 1NF`，对于每一个非平凡且非函数依赖的多值依赖：

```text
X ->-> Y
```

都有 `X` 含有码，则 `R ∈ 4NF`。

通俗判断：

```text
4NF 消除非平凡且非函数依赖的多值依赖。
```

关系：

```text
4NF 必然属于 BCNF
BCNF 不一定属于 4NF
```

解决多值依赖的常用分解：

```text
Teaching(C, T, B)
分解为：
CT(C, T)
CB(C, B)
```
