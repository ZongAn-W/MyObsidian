# 第七章 数据库设计：重要知识点总结

> 来源：《数据库系统概论（第6版）》第7章。重点掌握数据库设计步骤、需求分析、数据字典、E-R 模型、E-R 图集成、E-R 图向关系模型转换、逻辑结构优化、物理结构设计、数据库实施和维护。

## 1. 本章学习主线

数据库设计是数据库应用系统设计的重要组成部分，目标是在给定应用环境下设计出合适的数据库模式，并建立能有效存储和管理数据的数据库。

本章核心主线：

```text
现实世界需求
    -> 需求分析
    -> 概念结构设计（E-R 图）
    -> 逻辑结构设计（关系模式）
    -> 物理结构设计（存储、索引、聚簇）
    -> 数据库实施
    -> 运行和维护
```

考试复习时重点抓两类问题：

1. 数据库设计过程每一阶段做什么、产生什么结果。
2. 如何把 E-R 图转换为关系模型。

## 2. 数据库设计概述

### 2.1 数据库设计的含义

数据库设计是指对于一个给定的应用环境，构造合适的数据库模式、外模式和内模式，并据此建立数据库及其应用系统，使其能够有效地存储和管理数据，满足用户需求。

用户需求主要包括：

- 信息管理需求。
- 数据操作需求。
- 安全性和完整性需求。

### 2.2 数据库设计的目标

数据库设计的目标是为用户和各种应用系统提供信息基础设施和高效的运行环境。

高效运行环境通常包括：

- 较高的数据存取效率。
- 较高的存储空间利用率。
- 较高的数据库运行和维护效率。

### 2.3 数据库设计的特点

数据库设计的主要特点：

1. 三分技术，七分管理，十二分基础数据。
2. 数据库设计与应用系统设计紧密结合。
3. 结构设计和行为设计相结合。
4. 设计过程通常需要反复迭代。

理解：

- “基础数据”强调数据的收集、整理、组织和更新是设计成败的基础。
- 数据库设计不能只考虑表结构，还要考虑业务处理、应用程序、运行环境和维护。

## 3. 数据库设计的基本步骤

数据库设计通常分为 6 个阶段：

| 阶段 | 主要任务 | 主要结果 |
| --- | --- | --- |
| 需求分析 | 调查和分析用户需求 | 数据字典、用户需求说明 |
| 概念结构设计 | 抽象现实世界信息结构 | 概念模型、E-R 图 |
| 逻辑结构设计 | 把概念模型转换为数据模型 | 关系模式、外模式 |
| 物理结构设计 | 选择存储结构和存取方法 | 内模式、索引、聚簇等 |
| 数据库实施 | 建库、装载数据、编程调试 | 可运行数据库系统 |
| 运行和维护 | 监控、调整、备份恢复、重组重构 | 稳定运行的数据库 |

速记：

```text
需 -> 概 -> 逻 -> 物 -> 实 -> 维
```

### 3.1 各阶段形成的数据库模式

| 设计阶段 | 形成的模式 |
| --- | --- |
| 需求分析 | 用户需求、数据字典 |
| 概念结构设计 | 概念模式，通常用 E-R 图表示 |
| 逻辑结构设计 | DBMS 支持的数据模型，如关系模型 |
| 物理结构设计 | 内模式，如存储布局、索引、存取路径 |
| 逻辑结构设计后 | 可进一步设计外模式，即用户视图 |

## 4. 需求分析

### 4.1 需求分析的任务

需求分析是数据库设计的起点，任务是调查、收集和分析用户需求，明确系统要处理的数据和处理功能。

需求通常包括：

| 需求 | 含义 |
| --- | --- |
| 信息要求 | 用户需要从数据库获得哪些信息 |
| 处理要求 | 用户需要完成哪些数据处理功能，对性能有什么要求 |
| 安全性与完整性要求 | 数据访问控制、约束、一致性等要求 |

需求分析的结果是否准确，会直接影响后续设计是否合理。

### 4.2 需求分析的方法

常用调查方法：

- 跟班作业：亲身参与业务工作。
- 开调查会：与用户座谈。
- 请专人介绍：由熟悉业务的人讲解。
- 询问：对具体问题向相关人员提问。
- 问卷调查：设计表格由用户填写。
- 查阅记录：查看与系统有关的文档、报表、单据等。

常见分析方法：

- 结构化分析方法。
- 自顶向下、逐层分解。
- 建立数据字典和用户需求规格说明。

## 5. 数据字典

数据字典是需求分析阶段的重要成果，是对数据库中数据的描述，即元数据，而不是数据本身。

数据字典通常包括 5 类内容：

| 内容 | 含义 |
| --- | --- |
| 数据项 | 不可再分的数据单位 |
| 数据结构 | 数据项之间的组合关系 |
| 数据流 | 数据在系统中的传输路径 |
| 数据存储 | 数据结构停留或保存的地方 |
| 处理过程 | 对数据进行处理的逻辑 |

### 5.1 数据项

数据项是不可再分的数据单位。

描述内容通常包括：

```text
数据项名、含义说明、别名、数据类型、长度、取值范围、
取值含义、与其他数据项的逻辑关系
```

### 5.2 数据结构

数据结构反映数据项之间的组合关系。

描述内容：

```text
数据结构名、含义说明、组成
```

其中组成可以是数据项，也可以是其他数据结构。

### 5.3 数据流

数据流表示数据结构在系统中的传输路径。

描述内容通常包括：

```text
数据流名、说明、来源、去向、组成、平均流量、高峰期流量
```

### 5.4 数据存储

数据存储是数据结构停留或保存的地方。

描述内容通常包括：

```text
数据存储名、说明、编号、输入数据流、输出数据流、
组成、数据量、存取频度、存取方式
```

### 5.5 处理过程

处理过程说明具体处理逻辑。

描述内容通常包括：

```text
处理过程名、说明、输入、输出、处理
```

其中“处理”只需描述功能和处理要求，不一定写成程序。

## 6. 概念结构设计

### 6.1 概念结构设计的含义

概念结构设计是把需求分析得到的用户需求抽象为信息结构，即概念模型的过程。

概念模型的特点：

- 能真实、充分地反映现实世界。
- 易于理解，便于与用户交流。
- 易于修改和扩充。
- 易于向关系模型、网状模型、层次模型等数据模型转换。

常用工具：E-R 模型。

## 7. E-R 模型

### 7.1 E-R 模型的基本要素

E-R 模型包括：

| 要素 | 含义 |
| --- | --- |
| 实体 | 客观存在并可相互区别的事物 |
| 属性 | 实体或联系具有的特征 |
| 码 | 能唯一标识实体的属性或属性组 |
| 联系 | 实体之间的相互关系 |

### 7.2 E-R 图的表示

| 图形 | 表示对象 |
| --- | --- |
| 矩形 | 实体型 |
| 椭圆 | 属性 |
| 菱形 | 联系 |
| 无向边 | 连接实体、属性、联系 |

如果联系本身有属性，也要用椭圆与联系相连。

### 7.3 实体之间的联系类型

两个实体型之间的联系：

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| 1:1 | A 中一个实体最多对应 B 中一个实体，反之亦然 | 学院与正职院长 |
| 1:n | A 中一个实体对应 B 中多个实体，B 中一个实体对应 A 中一个实体 | 学院与学生 |
| m:n | A 中一个实体对应 B 中多个实体，B 中一个实体也对应 A 中多个实体 | 学生与课程 |

两个以上实体型之间也可以存在一对一、一对多、多对多联系。

单个实体型内部也可以存在联系，如课程与先修课程。

### 7.4 ISA 联系

ISA 联系表示“is a”语义，即父类和子类关系。

例子：

```text
研究生 is a 学生
本科生 is a 学生
```

特点：

- 子类继承父类的属性。
- 子类也可以有自己的属性。
- 可有不相交约束和可重叠约束。
- 可有完全特化和部分特化。

### 7.5 基数约束

基数约束用来细化参与联系的次数范围，常用 `min..max` 表示。

例子：

```text
0..1   可以不参加，也最多参加一次
1..1   必须参加且只能参加一次
0..*   可以不参加，也可以参加多次
20..30 必须参加 20 到 30 次
```

### 7.6 Part-of 联系

Part-of 联系表示“部分-整体”关系。

两种情况：

- 非独占的 Part-of：整体被破坏后，部分仍可独立存在。
- 独占的 Part-of：整体不存在时，部分也不能独立存在。

例子：

```text
汽车 - 轮子
```

若轮子可以拆下后继续独立存在，是非独占；若部分依赖整体存在，则是独占。

## 8. 用 E-R 图进行概念结构设计

### 8.1 实体与属性的划分原则

为了简化 E-R 图，现实世界中的事物能作为属性时尽量作为属性处理。

判断准则：

1. 属性不能再具有需要描述的性质，属性应是不可再分的数据项。
2. 属性不能与其他实体发生联系。

如果某个“属性”本身还有属性，或需要与其他实体发生联系，应把它建模为实体。

例子：

- 教师的“职称”如果只记录名称，可作为属性。
- 若需要记录职称代码、工资、岗位津贴、附加福利等，则“职称”更适合作为实体。

### 8.2 弱实体

若一个实体型的存在依赖于另一个实体型，则称为弱实体型。

弱实体通常不能仅靠自身属性形成码，需要依赖强实体的码和识别联系来标识。

## 9. E-R 图的集成

大型系统通常先设计各子系统的局部 E-R 图，再集成为全局 E-R 图。

集成通常分两步：

1. 合并：把各局部 E-R 图合并起来，生成初步 E-R 图。
2. 修改与重构：消除不必要的冗余，生成基本 E-R 图。

### 9.1 E-R 图集成中的冲突

| 冲突类型 | 含义 | 例子 |
| --- | --- | --- |
| 属性冲突 | 属性域、取值范围、单位、精度不同 | 出生日期精确到年/月/日不同 |
| 命名冲突 | 同名异义或异名同义 | “项目”在不同部门含义不同 |
| 结构冲突 | 同一对象在不同局部图中抽象不同 | 某处作实体，某处作属性 |

结构冲突还包括：

- 同一实体在不同局部 E-R 图中属性个数或排列次序不同。
- 同一联系在不同局部 E-R 图中联系类型不同。
- 同一对象在不同应用中抽象级别不同。

### 9.2 消除冗余

冗余包括：

- 冗余数据。
- 冗余联系。

消除冗余的方法：

- 根据数据字典分析。
- 根据数据流图分析。
- 使用规范化理论分析函数依赖，消除不必要的联系。

注意：并非所有冗余都必须删除，有时为了效率会保留部分冗余，但应明确其完整性约束和维护方式。

## 10. 逻辑结构设计

逻辑结构设计的任务是把概念结构设计得到的 E-R 图转换为具体 DBMS 支持的数据模型。本章重点是转换为关系模型。

## 11. E-R 图向关系模型的转换

转换目标：

```text
实体型、实体属性、实体间联系
    -> 关系模式、属性、码、外码
```

### 11.1 实体型转换

一个实体型通常转换为一个关系模式。

规则：

```text
实体的属性 -> 关系的属性
实体的码 -> 关系的码
```

### 11.2 1:1 联系转换

一对一联系有两种处理：

1. 转换为一个独立关系模式。
2. 与任意一端实体对应的关系模式合并。

若转换为独立关系模式：

- 两端实体的码以及联系自身属性作为该关系的属性。
- 任一端实体的码都可作为该关系的候选码。

若与某一端合并：

- 将另一端实体的码和联系自身属性加入该端关系模式。

### 11.3 1:n 联系转换

一对多联系有两种处理：

1. 转换为独立关系模式。
2. 与 n 端实体对应的关系模式合并。

若转换为独立关系模式：

- 两端实体的码和联系属性作为关系属性。
- n 端实体的码作为该关系的码。

若与 n 端合并：

- 将 1 端实体的码和联系属性加入 n 端关系模式。

### 11.4 m:n 联系转换

多对多联系必须转换为一个独立关系模式。

规则：

- 各实体的码和联系自身属性作为该关系的属性。
- 各实体码的组合通常作为该关系的码。

例子：

```text
学生(Sno, ...)
课程(Cno, ...)
选修(Sno, Cno, Grade)
```

其中 `(Sno, Cno)` 是 `选修` 的码，`Sno` 和 `Cno` 同时也是外码。

### 11.5 多元联系转换

三个或三个以上实体之间的多元联系通常转换为一个独立关系模式。

规则：

- 与联系相关的各实体的码加入该关系。
- 联系本身的属性加入该关系。
- 关系的码由各实体码组成，或由其中一部分组成，取决于联系语义。

### 11.6 具有相同码的关系模式

具有相同码的关系模式可以合并，以减少关系个数和连接操作。

## 12. 数据模型优化

逻辑结构设计得到的关系模式不一定最优，需要根据应用需求和规范化理论进行优化。

常用优化方法：

1. 确定数据依赖。
2. 对依赖进行极小化处理，消除冗余联系。
3. 按规范化理论分析关系模式，判断属于第几范式。
4. 根据处理要求判断是否需要合并或分解关系模式。
5. 适当进行水平分解或垂直分解。

### 12.1 水平分解

水平分解是把一个关系的元组分成若干子集合，每个子集合形成一个关系。

适用场景：

- 关系很大。
- 不同事务经常访问不同元组集合。
- 可以按业务范围、时间范围、地区等切分。

### 12.2 垂直分解

垂直分解是把一个关系的属性分成若干子集合，每个子集合形成一个关系。

适用场景：

- 某些属性经常一起被访问。
- 某些属性很少被访问。
- 希望减少连接或减少不必要的数据读取。

注意：垂直分解后应能通过公共码无损连接恢复原关系。

### 12.3 外模式设计

逻辑结构设计后，还应根据局部应用需求设计用户外模式。

外模式通常通过视图实现，作用包括：

- 使用更符合用户习惯的别名。
- 为不同级别用户定义不同视图。
- 简化复杂查询。
- 提供一定安全保护。

## 13. 物理结构设计

### 13.1 物理结构设计的含义

数据库的物理结构是数据在物理设备上的存储结构和存取方法。

物理结构设计通常分两步：

1. 确定数据库的物理结构。
2. 对物理结构进行评价。

评价重点：

- 时间效率。
- 空间效率。
- 维护代价。

### 13.2 物理结构设计的内容

主要内容：

- 选择关系模式的存取方法。
- 设计关系、索引等数据库文件的物理存储结构。
- 确定数据存放位置。
- 确定系统配置参数。
- 评价并调整物理结构。

### 13.3 选择存取方法

常用存取方法：

| 方法 | 作用 |
| --- | --- |
| 索引 | 提高按属性查找、连接、排序、分组的效率 |
| 聚簇 | 把经常一起访问的数据集中存放，减少 I/O |

B+ 树索引常用于范围查询和等值查询。

哈希索引常用于等值查询，通常不适合范围查询。

### 13.4 选择索引的常见依据

适合建立索引的情况：

- 经常出现在查询条件中的属性。
- 经常作为最大值、最小值等聚集函数参数的属性。
- 经常出现在连接条件中的属性。
- 经常用于排序、分组的属性。

不宜建立过多索引：

- 索引会占用存储空间。
- 插入、删除、更新时需要维护索引。
- 更新频繁的关系上索引过多会降低性能。

### 13.5 聚簇

聚簇是把某个或某些属性值相同的元组集中存放在连续物理块中，这些属性称为聚簇码。

适用情况：

- 经常按聚簇码查询。
- 经常按某属性范围访问大量记录。
- 多个关系经常按某些属性连接，可考虑聚簇存放。

注意：

- 一个数据库可建立多个聚簇。
- 一个关系通常只能加入一个聚簇。
- 维护聚簇代价较高。
- 聚簇码值应相对稳定。

## 14. 数据库实施和维护

### 14.1 数据库实施

数据库实施阶段包括：

1. 建立实际数据库结构。
2. 组织数据入库。
3. 编写和调试应用程序。
4. 数据库试运行。

数据装入时应注意：

- 数据来源可能分散，格式不一致。
- 需要进行数据转换、清洗和校验。
- 应防止不正确数据进入数据库。

### 14.2 数据库试运行

试运行是对数据库系统进行联合调试。

主要任务：

- 测试应用程序功能。
- 测试数据库操作是否正确。
- 测试性能指标是否达到设计目标。
- 发现问题后调整应用程序、物理结构或逻辑结构。

试运行阶段要做好数据库转储和恢复工作，防止故障造成数据破坏。

### 14.3 数据库运行和维护

数据库投入运行后，维护工作长期存在。

主要维护内容：

| 工作 | 含义 |
| --- | --- |
| 转储和恢复 | 制定备份计划，故障后恢复数据 |
| 安全性和完整性控制 | 根据应用变化调整权限和约束 |
| 性能监控、分析和改造 | 监控运行参数，优化系统性能 |
| 重组和重构 | 调整物理存储或修改模式 |

### 14.4 重组与重构

数据库重组：

- 不改变原设计的逻辑结构和物理结构。
- 重新安排存储位置，回收垃圾，减少碎片，提高性能。

数据库重构：

- 部分修改数据库模式或内模式。
- 例如增加/删除属性，改变数据项类型，增加/删除表或索引。

区别：

```text
重组：整理存储，不改模式
重构：修改模式或内模式
```

## 15. 易混概念辨析

### 15.1 概念结构、逻辑结构、物理结构

| 概念 | 面向对象 | 典型结果 |
| --- | --- | --- |
| 概念结构 | 现实世界和用户语义 | E-R 图 |
| 逻辑结构 | DBMS 支持的数据模型 | 关系模式 |
| 物理结构 | 存储设备和存取效率 | 索引、聚簇、存储布局 |

### 15.2 数据字典与数据库

| 概念 | 含义 |
| --- | --- |
| 数据字典 | 描述数据的数据，是元数据 |
| 数据库 | 实际存储和管理的数据集合 |

### 15.3 E-R 图与关系模式

| E-R 图元素 | 关系模型元素 |
| --- | --- |
| 实体型 | 关系模式 |
| 属性 | 属性 |
| 实体码 | 主码 |
| 联系 | 可转为独立关系，或并入相关实体关系 |

### 15.4 无损连接与消除冗余

分解关系模式时不能只追求减少冗余，还要考虑：

- 是否保持必要的数据依赖。
- 是否能无损连接恢复原关系。
- 是否满足应用性能要求。
