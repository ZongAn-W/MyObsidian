# 第四章 数据库安全性：重要知识点总结

> 来源：《数据库系统概论（第6版）》第4章。重点掌握数据库安全风险、安全控制技术、授权与回收、角色、强制访问控制、审计和数据加密。

## 1. 本章学习主线

数据库安全性是指保护数据库，防止因非法使用造成数据泄露、篡改或破坏。

本章主要解决三个问题：

1. 数据库为什么会不安全？
2. DBMS 如何控制用户访问？
3. 如何通过授权、角色、审计、加密等机制提高安全性？

核心记忆：

```text
数据库安全性 = 身份鉴别 + 存取控制 + 审计 + 加密 + 其他保护
```

## 5. 存取控制

存取控制的目标是保证用户只能访问被允许访问的数据，并只能执行被允许的操作。

存取控制机制通常包括两部分：

1. 定义用户权限。
2. 合法权限检查。

用户权限由两个因素组成：

- 数据库对象
- 操作类型

也就是说，权限不是孤立的，而是“用户能对哪些对象做哪些操作”。

## 6. 自主存取控制 DAC

### 6.1 自主存取控制的含义

自主存取控制（DAC）中，用户对不同数据库对象拥有不同权限，并且某些权限可以由授权用户授予其他用户。

SQL 标准主要通过：

```sql
GRANT
REVOKE
```

实现自主存取控制。

### 6.2 常见数据库对象和权限

常见对象：

- 数据库
- 模式
- 基本表
- 视图
- 索引
- 属性列

常见操作权限：

| 对象 | 常见权限 |
| --- | --- |
| 数据库/模式 | CREATE |
| 基本表 | CREATE TABLE、ALTER TABLE |
| 视图 | CREATE VIEW |
| 索引 | CREATE INDEX |
| 表或视图 | SELECT、INSERT、UPDATE、DELETE、REFERENCES |
| 属性列 | SELECT、INSERT、UPDATE、REFERENCES |

### 6.3 授权 GRANT

基本格式：

```sql
GRANT <权限> [, <权限>] ...
ON <对象类型> <对象名>
TO <用户> [, <用户>] ...
[WITH GRANT OPTION];
```

例子：

```sql
GRANT SELECT
ON TABLE Student
TO U1;
```

将多个权限授予用户：

```sql
GRANT SELECT, INSERT, UPDATE
ON TABLE Student
TO U1, U2;
```

授予某列权限：

```sql
GRANT UPDATE(Sno), SELECT
ON TABLE Student
TO U4;
```

授予传播权限：

```sql
GRANT INSERT
ON TABLE SC
TO U5
WITH GRANT OPTION;
```

`WITH GRANT OPTION` 表示被授权用户还可以把该权限再授予其他用户。

### 6.4 权限回收 REVOKE

基本格式：

```sql
REVOKE <权限> [, <权限>] ...
ON <对象类型> <对象名>
FROM <用户> [, <用户>] ...
[CASCADE | RESTRICT];
```

例子：

```sql
REVOKE SELECT
ON TABLE Student
FROM U1;
```

要点：

- 回收权限会影响用户后续访问。
- 若权限曾继续传播给其他用户，回收时可能涉及级联回收。
- 是否级联由系统和语句选项决定。

## 7. 数据库角色

### 7.1 角色的含义

角色是被命名的一组权限的集合。

使用角色可以简化权限管理：

- 先把权限授予角色。
- 再把角色授予用户。
- 用户通过角色获得一组权限。

### 7.2 创建角色

```sql
CREATE ROLE <角色名>;
```

例子：

```sql
CREATE ROLE R1;
```

### 7.3 给角色授权

```sql
GRANT SELECT, UPDATE, INSERT
ON TABLE Student
TO R1;
```

### 7.4 将角色授予用户或其他角色

```sql
GRANT <角色名>
TO <用户或角色>
[WITH ADMIN OPTION];
```

`WITH ADMIN OPTION` 表示获得该角色的用户或角色可以继续把该角色授予别人。

### 7.5 回收角色权限

```sql
REVOKE <权限>
ON <对象类型> <对象名>
FROM <角色名>;
```

角色的优点：

- 降低重复授权工作量。
- 便于按岗位管理权限。
- 便于统一调整一组用户的权限。

## 8. 强制存取控制 MAC

### 8.1 强制存取控制的含义

强制存取控制（MAC）不是由用户自主决定授权，而是系统根据安全级别和规则强制判断主体能否访问客体。

基本概念：

- 主体：用户、进程、事务等主动访问者。
- 客体：表、视图、元组、属性等被访问对象。
- 敏感度标记：主体和客体的安全级别。

### 8.2 常见安全级别

安全级别通常可以分为：

```text
绝密
机密
可信
公开
```

安全级别有高低之分，系统根据级别关系控制读写。

### 8.3 MAC 的基本思想

强制存取控制通常遵循类似以下原则：

- 主体只能读取安全级别不高于自己级别的客体。
- 主体只能写入安全级别不低于自己级别的客体。

通俗记忆：

```text
向下读，向上写
```

目的：

- 防止低级别主体读取高级别数据。
- 防止高级别数据被写入低级别对象造成泄露。

## 9. 视图机制

视图也可以作为安全控制手段。

做法：

1. 为不同用户定义不同视图。
2. 只把视图权限授予用户。
3. 用户只能通过视图看到允许访问的数据。

优点：

- 隐藏敏感列。
- 隐藏敏感行。
- 简化授权。
- 提供一定程度的数据隔离。

例子：只给用户开放“学生姓名、课程名、成绩”视图，而不开放学生身份证号、家庭地址等敏感列。

## 10. 审计

### 10.1 审计的含义

审计是把用户对数据库的操作记录到审计日志中，以便事后检查、分析和追踪。

审计记录通常包括：

- 操作用户
- 操作时间
- 操作对象
- 操作类型
- 操作结果

### 10.2 审计的作用

审计可以：

- 发现可疑访问。
- 追踪攻击来源。
- 约束用户行为。
- 为安全事故分析提供证据。

### 10.3 AUDIT 与 NOAUDIT

开启审计：

```sql
AUDIT <操作> ON <对象>;
```

关闭审计：

```sql
NOAUDIT <操作> ON <对象>;
```

审计设置和审计日志通常保存在数据字典或系统审计表中。

## 11. 数据加密

数据加密是保护数据安全的重要技术，尤其适用于高度敏感数据。

加密基本思想：

```text
明文 → 加密算法 → 密文
```

不知道解密方法或密钥的人无法直接获得数据内容。

### 11.1 存储加密

存储加密保护数据在磁盘上的安全。

常见方式：

- 透明存储加密：由 DBMS 自动加密和解密，对用户和应用程序透明。
- 非透明存储加密：通过加密函数实现，应用需要显式处理加密和解密。

### 11.2 传输加密

传输加密保护数据在网络传输过程中的安全。

常见方式：

- 链路加密：对链路层传输数据加密，通常报文和报头都加密。
- 端到端加密：在发送端加密、接收端解密，通常只加密报文内容。

数据库系统中常见的传输保护方案可以基于 SSL 等安全协议。

## 12. 其他安全性保护

除 DBMS 内部机制外，数据库安全还依赖整体安全环境。

包括：

- 操作系统安全。
- 网络安全。
- 物理安全。
- 应用程序安全。
- 数据备份与恢复。
- 管理制度与审计制度。

数据库安全不是单点技术，而是多层防护体系。

## 13. 易混概念辨析

### 13.1 安全性与完整性

| 概念 | 关注点 |
| --- | --- |
| 安全性 | 防止非法访问、泄露、篡改和破坏 |
| 完整性 | 保证数据正确、有效、相容 |

### 13.2 GRANT OPTION 与 ADMIN OPTION

| 选项 | 用于 | 含义 |
| --- | --- | --- |
| WITH GRANT OPTION | 对象权限 | 用户可继续授予该对象权限 |
| WITH ADMIN OPTION | 角色 | 用户或角色可继续授予该角色 |

### 13.3 自主存取控制与强制存取控制

| 类型 | 控制依据 | 特点 |
| --- | --- | --- |
| DAC | 用户授权 | 灵活，可传播权限 |
| MAC | 安全级别和强制规则 | 更严格，用户不能随意改变 |

### 13.4 审计与加密

| 技术 | 作用 |
| --- | --- |
| 审计 | 记录和追踪操作 |
| 加密 | 防止数据被直接读懂 |
