本章主线：

```text

```

## 4. 数据库恢复概述

数据库恢复：当数据库被破坏或处于不一致状态时，将数据库恢复到某个已知正确状态的过程。

导致恢复的原因：

- 硬件故障；
- 软件错误；
- 操作员误操作；
- 事务异常终止；
- 系统断电；
- 磁盘损坏；
- 恶意破坏。

恢复机制的核心思想：

```text
用冗余数据恢复正确数据
```

常见冗余数据：

- 后备副本；
- 日志文件；
- 数据库镜像。

## 9. UNDO 和 REDO

### 9.1 UNDO

UNDO：撤销事务已经做过的修改，使数据库恢复到事务执行前状态。

依赖信息：

- 更新前旧值。

常用于：

- 事务故障；
- 系统故障中未提交事务。

例：

```text
日志：T1 将 A 从 100 改为 80
UNDO：把 A 恢复为 100
```

### 9.2 REDO

REDO：重新执行已经提交事务的修改，确保提交结果写入数据库。

依赖信息：

- 更新后新值。

常用于：

- 系统故障中已提交但未完全写盘的事务；
- 介质故障中装入后备副本后的恢复。

例：

```text
日志：T2 将 B 从 50 改为 70，且 T2 已提交
REDO：把 B 写成 70
```

### 9.3 UNDO 与 REDO 对比

| 项目 | UNDO | REDO |
|---|---|---|
| 中文 | 撤销 | 重做 |
| 目标 | 消除未完成事务影响 | 保证已提交事务结果存在 |
| 使用值 | 更新前旧值 | 更新后新值 |
| 常见对象 | 未提交事务 | 已提交事务 |
| 日志扫描 | 常反向扫描 | 常正向扫描 |

记忆：

```text
没提交的不能留：UNDO
已提交的不能丢：REDO
```

## 10. 恢复策略

### 10.1 事务故障恢复

事务故障发生时，该事务未到达正常终点。

恢复步骤：

1. 反向扫描日志文件，查找该事务的更新操作。
2. 对每个更新执行逆操作。
3. 继续反向扫描，直到该事务的开始标记。

本质：

```text
事务故障 -> UNDO 该事务
```

### 10.2 系统故障恢复

系统故障会导致：

- 未完成事务的修改可能已经写入数据库；
- 已提交事务的修改可能还停留在缓冲区中。

恢复步骤：

1. 正向扫描日志。
2. 将故障发生时未提交事务加入 `UNDO-LIST`。
3. 将故障发生前已提交事务加入 `REDO-LIST`。
4. 对 `UNDO-LIST` 中事务执行 UNDO。
5. 对 `REDO-LIST` 中事务执行 REDO。

核心：

```text
未完成事务 -> UNDO
已提交事务 -> REDO
```

### 10.3 介质故障恢复

介质故障导致数据库或日志文件所在介质损坏。

恢复步骤：

1. 装入最近一次数据库后备副本；
2. 装入必要的日志文件副本；
3. 根据日志重做自转储以来已提交事务；
4. 必要时撤销未完成事务；
5. 恢复到故障前某个一致状态。

核心：

```text
介质故障 -> 先恢复副本，再用日志追赶
```

## 11. 具有检查点的恢复技术

### 11.1 为什么需要检查点

如果没有检查点，恢复时可能要从头扫描大量日志。

问题：

- 日志很长，搜索耗时；
- 很多已提交事务的结果早已写入数据库；
- 重做这些事务浪费时间。

检查点的作用：

```text
缩短恢复时需要扫描的日志范围，提高恢复效率。
```

### 11.2 检查点记录的内容

检查点记录通常包括：

- 建立检查点时所有正在执行的事务清单；
- 这些事务最近一个日志记录的地址。

系统还维护重新开始文件，用于记录各检查点在日志文件中的地址。

### 11.3 动态维护检查点的一般步骤

建立检查点时，系统通常：

1. 将当前日志缓冲区中的日志记录写入磁盘日志文件；
2. 在日志文件中写入检查点记录；
3. 将当前数据缓冲区中的数据页写入磁盘；
4. 把检查点记录在日志文件中的地址写入重新开始文件。

注意：

- 检查点不一定要求停止所有事务；
- 它使系统知道从哪里开始恢复更合适。

### 11.4 根据检查点判断 UNDO/REDO

设：

- `Tc`：检查点时刻；
- `Tf`：系统故障时刻。

常见情况：

| 事务情况 | 恢复处理 |
|---|---|
| 检查点前已提交 | 通常不需要 REDO |
| 检查点前开始，检查点后、故障前提交 | 需要 REDO |
| 检查点前开始，故障时未提交 | 需要 UNDO |
| 检查点后开始，故障前提交 | 需要 REDO |
| 检查点后开始，故障时未提交 | 需要 UNDO |

快速判断：

```text
故障时没提交 -> UNDO
故障前已提交但可能未写盘 -> REDO
检查点前已提交且数据已写盘 -> 通常不用 REDO
```

### 11.5 检查点恢复步骤

使用检查点恢复时：

1. 从重新开始文件找到最后一个检查点记录的位置。
2. 由检查点记录得到当时正在执行的事务清单。
3. 建立 `UNDO-LIST` 和 `REDO-LIST`。
4. 从检查点开始扫描后续日志。
5. 新开始事务先放入 `UNDO-LIST`。
6. 遇到提交记录，将事务从 `UNDO-LIST` 移到 `REDO-LIST`。
7. 日志扫描结束后，对 `UNDO-LIST` 执行 UNDO，对 `REDO-LIST` 执行 REDO。

## 12. 数据库镜像

### 12.1 数据库镜像定义

数据库镜像是 DBMS 自动将整个数据库或关键数据复制到另一个磁盘上的技术。

当主数据库更新时，系统自动把更新复制到镜像数据库，以保持两者一致。

### 12.2 数据库镜像的作用

主要作用：

- 提高可用性；
- 介质故障时可用镜像继续提供服务；
- 缩短恢复时间；
- 减轻 DBA 手工恢复负担；
- 可让读操作访问镜像数据，缓解主库压力。

### 12.3 数据库镜像的缺点

缺点：

- 需要额外存储空间；
- 数据复制会降低更新效率；
- 系统维护成本较高。

因此，实际系统中常只对关键数据和日志文件建立镜像，而不一定镜像整个数据库。

## 13. 高频考点

### 13.1 简答题

**事务的四个 ACID 特性是什么？**

答：

```text
原子性、一致性、隔离性、持续性。
```

**数据库恢复技术能保证事务的哪些特性？**

答：

```text
主要保证原子性和持续性；
一致性也依赖恢复机制维持；
隔离性主要由并发控制保证。
```

**为什么事务故障要执行 UNDO？**

答：事务未提交，不能让它的部分修改保留在数据库中，必须撤销其已完成的更新，保证原子性。

**为什么系统故障既要 UNDO 又要 REDO？**

答：故障时未提交事务的修改可能已经写入数据库，需要 UNDO；已提交事务的修改可能还在缓冲区未写入磁盘，需要 REDO。

**为什么动态转储必须配合日志？**

答：动态转储期间事务仍在运行，后备副本可能不是一致状态，必须利用日志撤销或重做转储期间的更新，使数据库恢复到一致状态。

### 13.2 判断题

常见陷阱：

- “事务提交后，修改一定已经写入磁盘。”  
  错。可能还在缓冲区，但系统必须保证可通过日志恢复。

- “只要有后备副本，就不需要日志。”  
  错。后备副本只能恢复到转储时刻，之后的已提交更新需要日志重做。

- “系统故障一定破坏数据库物理文件。”  
  错。系统故障通常不破坏磁盘上的数据库文件，介质故障才会破坏外存数据。

- “检查点是为了防止故障发生。”  
  错。检查点不能防止故障，只能提高故障后的恢复效率。

### 13.3 计算/分析题

常见题型：

给出日志序列和故障发生位置，判断哪些事务要 UNDO，哪些要 REDO。

解题套路：

```text
1. 找故障发生时刻。
2. 看每个事务是否有 BEGIN。
3. 看故障前是否有 COMMIT。
4. 有 BEGIN 无 COMMIT：UNDO。
5. 有 COMMIT：REDO。
6. 若有检查点，从最后一个检查点附近开始分析。
```

## 14. 易混概念辨析

| 易混点 | 正确认识 |
|---|---|
| COMMIT vs 写磁盘 | COMMIT 表示事务提交，不等于所有数据页已写盘 |
| ROLLBACK vs UNDO | ROLLBACK 是事务回滚命令，UNDO 是恢复中的撤销操作 |
| REDO vs 重新执行 SQL | REDO 通常按日志中的更新后值重做，不一定重新执行原 SQL |
| 系统故障 vs 介质故障 | 系统故障多是内存状态丢失，介质故障是外存数据损坏 |
| 静态转储 vs 动态转储 | 静态转储停事务，动态转储允许事务运行 |
| 海量转储 vs 增量转储 | 海量备份全库，增量只备份变化部分 |
| 日志 vs 后备副本 | 日志记录变化，后备副本保存某时刻数据状态 |
| 检查点 vs 备份 | 检查点是日志恢复辅助点，不是完整备份 |
| 数据库镜像 vs 数据转储 | 镜像实时或近实时复制，转储是定期备份 |
