# 第一章 绪论：重要知识点总结

> 来源：《数据库系统概论（第6版）》第1章。适合用于课前预习、期末复习和后续章节概念串联。

## 1. 本章学习主线

第一章主要建立数据库课程的基本概念框架：

1. 先理解数据库系统的四个基本概念：数据、数据库、数据库管理系统、数据库系统。
2. 再了解数据管理技术的发展：人工管理阶段、文件系统阶段、数据库系统阶段。
3. 接着掌握数据模型：概念模型、数据模型三要素、层次模型、网状模型、关系模型。
4. 最后理解数据库系统的结构：三级模式、两级映像，以及数据库系统的组成和体系结构。

本章的核心思想是：数据库系统把信息系统从“以程序为中心”推进到“以共享数据为中心”。

## 2. 数据库系统的四个基本概念

### 2.1 数据

数据是描述事物的符号记录，是数据库中存储的基本对象。

数据不只包括数字，还包括文字、图形、图像、音频、视频、聊天记录、网购记录、病历等。数据必须结合语义才能表达真实含义，例如“93”可能表示成绩、体重，也可能表示人数。

要点：

- 数据 = 符号记录 + 语义解释。
- 记录是计算机中表示和存储数据的一种格式。
- 现代数据库处理的数据类型越来越丰富，已经不限于传统数值型数据。

### 2.2 数据库

数据库（DB）是长期存储在计算机内、有组织、可共享的大量数据的集合。

数据库中的数据按一定的数据模型组织、描述和存储，具有以下特征：

- 长期存储在计算机中。
- 有组织，而不是杂乱文件堆积。
- 可被多个用户或多个应用共享。
- 数据冗余较小。
- 数据独立性较高。
- 可扩展性较强。

### 2.3 数据库管理系统

数据库管理系统（DBMS）是位于用户和操作系统之间的数据管理软件，是数据库系统的核心软件。

DBMS 的主要功能：

- 数据定义：提供 DDL，定义数据库对象的结构。
- 数据组织、存储和管理：管理数据字典、用户数据、存取路径等。
- 数据操纵：提供 DML，实现查询、插入、删除、修改等操作。
- 事务管理和运行管理：保证事务正确执行，支持安全性、完整性、并发控制和故障恢复。
- 数据库建立和维护：包括初始数据装入、转换、转储、恢复、重组、性能监视等。
- 其他功能：通信、数据转换、异构数据库互访等。

### 2.4 数据库系统

数据库系统（DBS）是引入数据库后的计算机系统，通常由以下部分组成：

- 数据库（DB）
- 数据库管理系统（DBMS）及应用开发工具
- 应用系统
- 数据库管理员（DBA）

关系可以理解为：

```text
用户 / 应用系统
       ↓
数据库管理系统 DBMS
       ↓
数据库 DB
```

日常语境中有时会把“数据库系统”简称为“数据库”，但严格来说二者不同。

## 3. 数据管理技术的发展

数据管理技术经历了三个阶段：人工管理阶段、文件系统阶段、数据库系统阶段。

### 3.1 人工管理阶段

特点：

- 主要用于科学计算。
- 数据不长期保存。
- 数据由应用程序自己管理。
- 数据不共享，冗余度高。
- 数据不具有独立性，数据结构变化会导致应用程序修改。

本质问题：数据依附于程序，程序和数据高度耦合。

### 3.2 文件系统阶段

特点：

- 数据可以长期保存在外存。
- 操作系统提供文件系统来管理数据。
- 文件内的记录具有一定结构。
- 数据仍然面向具体应用。

主要缺点：

- 数据共享性差，冗余度高。
- 数据容易不一致。
- 数据独立性弱。
- 文件之间孤立，难以反映现实世界中事物之间的整体联系。

### 3.3 数据库系统阶段

数据库系统阶段的主要特征：

- 整体数据结构化。
- 数据共享性高，冗余度低，易扩充。
- 数据独立性强。
- 数据由 DBMS 统一管理和控制。

DBMS 统一控制的重点：

- 安全性：防止非法使用造成数据泄露或破坏。
- 完整性：保证数据正确、有效、相容。
- 并发控制：协调多用户同时访问和修改数据。
- 恢复：发生故障后把数据库恢复到正确状态。

数据库系统的意义：

- 从以程序为中心转向以数据为中心。
- 便于数据集中管理。
- 简化应用系统开发与维护。
- 提高数据利用率和决策可靠性。

## 4. 数据模型

### 4.1 数据模型的含义

模型是对现实世界对象特征的抽象。数据模型是对现实世界数据特征的抽象，用来描述数据、组织数据并对数据进行操作。

数据模型是数据库系统的核心和基础。现有数据库系统都是基于某种数据模型建立的。

### 4.2 数据建模过程

数据库中的数据建模通常分两步：

1. 建立概念模型：把现实世界抽象为信息世界，从用户角度描述数据。
2. 转换为数据模型：把概念模型转换为某个 DBMS 支持的数据模型，进入机器世界。

可以记为：

```text
现实世界 → 信息世界（概念模型）→ 机器世界（DBMS 支持的数据模型）
```

### 4.3 概念模型

概念模型用于信息世界建模，是数据库设计人员与用户交流的工具。

信息世界中的基本概念：

- 实体：客观存在且可相互区别的事物，如学生、课程、教师。
- 属性：实体具有的某一特性，如学生的学号、姓名、性别。
- 码：唯一标识实体的属性集，如学号。
- 实体型：用实体名和属性名集合描述同类实体，如学生（学号，姓名，性别，出生日期，主修专业）。
- 实体集：同一类型实体的集合，如全体学生。
- 联系：实体内部或实体之间的关联。

实体之间常见联系：

- 一对一（1:1）
- 一对多（1:n）
- 多对多（m:n）

常用的概念模型表示方法是实体-联系模型，即 E-R 模型。E-R 图通常用来表示实体、属性和实体之间的联系。

### 4.4 数据模型的三要素

数据模型由三部分组成：

1. 数据结构：描述数据库的组成对象以及对象之间的联系，是数据模型最重要的方面，描述系统静态特性。
2. 数据操纵：描述允许对数据库对象执行的操作，如查询、插入、删除、修改，描述系统动态特性。
3. 完整性约束：规定数据及其联系必须满足的约束，用于保证数据正确、有效、相容。

一句话记忆：

```text
数据结构管“有什么”，数据操纵管“能做什么”，完整性约束管“必须满足什么规则”。
```

## 5. 常见数据模型

### 5.1 层次模型

层次模型用树形结构表示实体及实体之间的联系。

结构特点：

- 有且只有一个根结点没有双亲结点。
- 根以外的其他结点有且只有一个双亲结点。
- 主要表示一对多联系。

优点：

- 结构简单清晰。
- 查询效率较高。
- 对层次型关系有较好的完整性支持。

缺点：

- 难以自然表示多对多联系。
- 查询子女结点必须经过双亲结点。
- 结构限制较强，应用程序编写复杂。

适用理解：层次模型像一棵倒置的树，适合描述天然具有层级关系的对象。

### 5.2 网状模型

网状模型允许更复杂的记录联系。

结构特点：

- 可以有多个结点没有双亲结点。
- 一个结点可以有多个双亲结点。
- 允许两个结点之间存在多种联系。
- 层次模型可以看作网状模型的特例。

优点：

- 能更直接地描述现实世界中的复杂联系。
- 性能较好，存取效率较高。

缺点：

- 结构复杂，不利于普通用户掌握。
- 数据定义语言和数据操纵语言复杂。
- 应用程序需要了解存取路径，数据独立性较弱。

### 5.3 关系模型

关系模型用二维表表示实体及实体之间的联系，是目前最重要、最常用的数据模型。

基本术语：

- 关系：一张二维表。
- 元组：表中的一行。
- 属性：表中的一列。
- 码：能唯一标识元组的属性或属性组。
- 域：属性的取值范围。
- 关系模式：对关系结构的描述。

优点：

- 建立在严格的数学概念基础上。
- 概念单一，实体和联系都用关系表示。
- 数据独立性较高。
- 查询语言非过程化，用户只需说明“要什么”，不必说明“怎么取”。

关系模型是后续学习 SQL、关系代数、关系数据库设计的基础。

## 6. 数据库系统的结构

### 6.1 三级模式结构

数据库系统通常采用三级模式结构：

1. 外模式：用户视图，是用户能看到和使用的局部数据逻辑结构。
2. 模式：数据库中全体数据的逻辑结构和特征描述，也称逻辑模式。
3. 内模式：数据物理结构和存储方式的描述，也称存储模式。

三者关系：

```text
外模式：面向用户，多个
模式：面向全局逻辑结构，只有一个
内模式：面向物理存储，只有一个
```

### 6.2 两级映像与数据独立性

两级映像：

- 外模式/模式映像
- 模式/内模式映像

它们保证数据库系统的数据独立性：

- 逻辑独立性：模式改变时，外模式和应用程序尽量不变。
- 物理独立性：内模式改变时，模式和应用程序尽量不变。

记忆：

```text
外/模映像保证逻辑独立性。
模/内映像保证物理独立性。
```

## 7. 数据库系统的组成

数据库系统通常由硬件、软件、数据库和人员组成。

### 7.1 硬件平台

包括 CPU、内存、外存、输入输出设备、网络设备等。数据库系统对硬件通常有较高要求，尤其是存储容量、I/O 能力和可靠性。

### 7.2 软件

主要包括：

- DBMS
- 操作系统
- 应用开发工具
- 应用系统

DBMS 是其中最核心的软件。

### 7.3 数据库

数据库是系统管理和使用的数据资源，是数据库系统的核心对象。

### 7.4 人员

数据库系统相关人员包括：

- 数据库管理员（DBA）：负责数据库规划、设计、运行、维护、安全和性能管理。
- 系统分析员和数据库设计人员：负责需求分析和数据库设计。
- 应用程序员：开发使用数据库的应用系统。
- 最终用户：通过应用系统或查询语言使用数据库。

## 8. 数据库系统的体系结构

从用户使用方式看，数据库系统可以有不同体系结构。

常见结构：

- 集中式结构：数据库和 DBMS 集中运行在一台主机或服务器上。
- 客户/服务器结构（C/S）：客户端负责用户界面和部分处理，服务器负责数据库管理。
- 浏览器/服务器结构（B/S）：用户通过浏览器访问应用服务器，再访问数据库服务器。
- 分布式结构：数据分布在多个站点，由分布式数据库系统统一管理。

理解重点：体系结构关注数据库系统在硬件、软件和网络环境中的部署方式。

## 9. 易混概念辨析

### 9.1 数据、数据库、DBMS、DBS

| 概念 | 含义 | 关键词 |
| --- | --- | --- |
| 数据 | 描述事物的符号记录 | 符号 + 语义 |
| 数据库 DB | 有组织、可共享的大量数据集合 | 数据集合 |
| 数据库管理系统 DBMS | 管理数据库的软件 | 管理软件 |
| 数据库系统 DBS | 引入数据库后的完整计算机系统 | DB + DBMS + 应用 + 人员 |

### 9.2 文件系统与数据库系统

| 比较项 | 文件系统 | 数据库系统 |
| --- | --- | --- |
| 数据组织 | 文件内有结构，整体结构弱 | 整体结构化 |
| 数据共享 | 共享性弱 | 共享性强 |
| 数据冗余 | 冗余度高 | 冗余度低 |
| 数据独立性 | 弱 | 强 |
| 数据控制 | 主要由应用程序控制 | 由 DBMS 统一控制 |

### 9.3 逻辑独立性与物理独立性

| 类型 | 改变对象 | 尽量不受影响的对象 |
| --- | --- | --- |
| 逻辑独立性 | 数据库逻辑结构 | 外模式、应用程序 |
| 物理独立性 | 数据物理存储结构 | 模式、应用程序 |
