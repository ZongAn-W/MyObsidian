# 2024-2025 数据库原理期末试卷答案

> 依据本知识图谱中的关系代数、结构化查询语言、视图、完整性约束、函数依赖、范式规范化、实体-联系模型、实体-联系图向关系模型转换、自主存取控制等知识点整理。

## 第一题：关系代数

已知：

```
学生(学号, 姓名, 性别, 年龄)
课程(课程号, 课程名, 授课教师)
选课(学号, 课程号, 成绩)
```

### 解题讲解

这一题考关系代数。关系代数的基本套路是：先用选择 `σ` 过滤行，再用连接 `⋈` 把相关关系合起来，最后用投影 `π` 取题目要求输出的属性。

这里三个关系的含义很清楚：学生关系保存学生基本信息，课程关系保存课程基本信息，选课关系连接学生和课程。凡是题目同时涉及学生信息和课程信息，一般都要通过 `选课` 把 `学生` 和 `课程` 连接起来。

第 1 小问要输出学号和姓名，但筛选条件是课程名为“数学”。课程名在 `课程` 中，学号在 `选课` 中，姓名在 `学生` 中，所以要把三个关系连接起来。最后只投影 `学号, 姓名`，不要把成绩、课程号等多余属性带出来。

第 2 小问只问学号，条件是授课教师为“程军”。教师名在 `课程` 中，选了该课程的学生学号在 `选课` 中，所以连接 `选课` 和筛选后的 `课程` 即可，不需要连接 `学生`。

第 3 小问是“同时选修 1 和 3”，关键词是“同时”。可以先分别找出选修课程 `1` 的学号集合和选修课程 `3` 的学号集合，再取交集。交集表示两个条件都满足。

第 4 小问是“数据库原理或操作系统”，关键词是“或”。可以在选择条件中用“或”，也可以把两个查询结果做并集。并集表示满足任意一个条件即可。

第 5 小问只涉及学生基本信息，所有条件和输出属性都在 `学生` 关系里，所以不需要连接。先选择性别和年龄范围，再投影学号、姓名、年龄。

第 6 小问出现“全部课程”，这是关系代数中典型的除法题。`π学号,课程号(选课)` 表示每个学生实际选了哪些课，`π课程号(课程)` 表示所有课程。用前者除以后者，得到“选完所有课程的学号”，再与 `学生` 连接取姓名。

第 7 小问是“选修了 1042 号同学选修的全部课程”。这里的“全部”不是全部课程，而是 `1042` 号同学选过的全部课程。因此除数要写成 `π课程号(σ学号='1042'(选课))`，表示 1042 号同学的课程集合。

1. 查询选修了 `数学` 课程的学生学号和姓名：

```
π学号,姓名(学生 ⋈ 选课 ⋈ σ课程名='数学'(课程))
```

2. 查询选修了 `程军` 老师课程的学生学号：

```
π学号(选课 ⋈ σ授课教师='程军'(课程))
```

3. 查询同时选修课程号 `1` 和课程号 `3` 的学生学号：

```
π学号(σ课程号='1'(选课)) ∩ π学号(σ课程号='3'(选课))
```

4. 查询选修了 `数据库原理` 或 `操作系统` 的学生学号：

```
π学号(选课 ⋈ σ课程名='数据库原理' ∨ 课程名='操作系统'(课程))
```

也可写为：

```
π学号(选课 ⋈ σ课程名='数据库原理'(课程))
∪
π学号(选课 ⋈ σ课程名='操作系统'(课程))
```

5. 查询年龄在 `[18, 20]` 之间的女学生学号、姓名、年龄：

```
π学号,姓名,年龄(σ性别='女' ∧ 年龄>=18 ∧ 年龄<=20(学生))
```

6. 查询选修了全部课程的学生姓名：

```
π姓名(学生 ⋈ (π学号,课程号(选课) ÷ π课程号(课程)))
```

7. 查询选修了 `1042` 号同学选修的全部课程的学生学号：

```
π学号(π学号,课程号(选课) ÷ π课程号(σ学号='1042'(选课)))
```

## 第二题：结构化查询语言

已知：

```
职工(职工号, 姓名, 性别, 年龄)
社团(社团编号, 社团名称, 负责人, 活动地点)
参加(职工号, 社团编号, 参加日期)
```

### 解题讲解

这一题综合考结构化查询语言的数据定义、视图、连接查询、嵌套查询、聚集查询和完整性约束。

第 1 小问是建表。建表时要先判断主键和外键。`职工` 的主键是 `职工号`，`社团` 的主键是 `社团编号`。`社团` 中的 `负责人` 是某个职工，所以它是外键，参照 `职工(职工号)`。`参加` 是职工和社团之间的联系表，因此同时包含 `职工号` 和 `社团编号` 两个外键。通常可把 `(职工号, 社团编号)` 作为主键，表示某个职工参加某个社团只记录一次；如果允许同一职工在不同日期多次参加同一社团，则把 `参加日期` 也加入主键。

第 2 小问是创建视图。视图本质上是从基本表导出的虚表。`负责人详情视图` 需要社团编号、社团名称、负责人姓名、负责人性别，所以要把 `社团` 与 `职工` 按 `负责人 = 职工号` 连接。`参加人员详情视图` 同时需要职工信息、社团信息和参加日期，所以要把 `职工`、`社团`、`参加` 三张表连接起来。

第 3 小问要查参加“篮球社”或“舞蹈团”的职工号。社团名称在 `社团` 表里，职工号在 `参加` 表里，所以先连接，再用 `IN ('篮球社', '舞蹈团')` 表示二者之一。`DISTINCT` 用来去掉同一职工可能重复出现的情况。

第 4 小问要查“没有参加任何社团”的职工。这里适合用 `NOT EXISTS`：对某个职工来说，如果不存在任何一条参加记录与他的职工号相同，那么这个职工就是没有参加社团的人。

第 5 小问是“参加了全部社团”，在结构化查询语言里通常用双重 `NOT EXISTS` 表达“全部”。外层意思是：不存在一个社团，使得该职工没有参加它。换句话说，对所有社团，该职工都参加了。

第 6 小问是“至少参加了一个 1001 号职工参加过的社团”。先用子查询找出 `1001` 号职工参加过的社团编号集合，再查哪些职工的参加记录落在这个集合中。若题目要求“和 1001 一起参加过的其他职工”，就要排除 `1001` 本人。

第 7 小问是统计每个社团参加人数。为了让没有人参加的社团也能显示出来，答案从 `社团` 表出发，对每个社团用相关子查询统计 `参加` 表中对应的不同职工号个数。这样比只对 `参加` 表分组更完整。

1. 建表语句：

```
CREATE TABLE "职工" (
    "职工号" CHAR(8) PRIMARY KEY,
    "姓名" VARCHAR(20),
    "性别" CHAR(1),
    "年龄" INT
);

CREATE TABLE "社团" (
    "社团编号" CHAR(8) PRIMARY KEY,
    "社团名称" VARCHAR(30),
    "负责人" CHAR(8),
    "活动地点" VARCHAR(50),
    FOREIGN KEY ("负责人") REFERENCES "职工"("职工号")
);

CREATE TABLE "参加" (
    "职工号" CHAR(8),
    "社团编号" CHAR(8),
    "参加日期" DATE,
    PRIMARY KEY ("职工号", "社团编号"),
    FOREIGN KEY ("职工号") REFERENCES "职工"("职工号"),
    FOREIGN KEY ("社团编号") REFERENCES "社团"("社团编号")
);
```

如果题目认为同一职工可在不同日期多次参加同一社团，则 `参加` 的主键也可改为：

```
PRIMARY KEY ("职工号", "社团编号", "参加日期")
```

2. 创建视图：

```
CREATE VIEW "负责人详情视图"("社团编号", "社团名称", "负责人姓名", "负责人性别")
AS
SELECT "社团表"."社团编号", "社团表"."社团名称", "职工表"."姓名", "职工表"."性别"
FROM "社团" "社团表", "职工" "职工表"
WHERE "社团表"."负责人" = "职工表"."职工号";

CREATE VIEW "参加人员详情视图"("职工号", "职工姓名", "社团编号", "社团名称", "参加日期")
AS
SELECT "职工表"."职工号", "职工表"."姓名", "社团表"."社团编号", "社团表"."社团名称", "参加表"."参加日期"
FROM "职工" "职工表", "社团" "社团表", "参加" "参加表"
WHERE "职工表"."职工号" = "参加表"."职工号"
  AND "社团表"."社团编号" = "参加表"."社团编号";
```

3. 查询参加了 `篮球社` 或 `舞蹈团` 的职工号：

```
SELECT DISTINCT "参加表"."职工号"
FROM "参加" "参加表", "社团" "社团表"
WHERE "参加表"."社团编号" = "社团表"."社团编号"
  AND "社团表"."社团名称" IN ('篮球社', '舞蹈团');
```

4. 查询没有参加任何社团的职工详情：

```
SELECT *
FROM "职工" "职工表"
WHERE NOT EXISTS (
    SELECT *
    FROM "参加" "参加表"
    WHERE "参加表"."职工号" = "职工表"."职工号"
);
```

5. 查询参加了全部社团的职工详情：

```
SELECT *
FROM "职工" "职工表"
WHERE NOT EXISTS (
    SELECT *
    FROM "社团" "社团表"
    WHERE NOT EXISTS (
        SELECT *
        FROM "参加" "参加表"
        WHERE "参加表"."职工号" = "职工表"."职工号"
          AND "参加表"."社团编号" = "社团表"."社团编号"
    )
);
```

6. 查询至少参加了一个 `1001` 号职工参加过的社团的职工号：

```
SELECT DISTINCT "参加表1"."职工号"
FROM "参加" "参加表1"
WHERE "参加表1"."社团编号" IN (
    SELECT "参加表2"."社团编号"
    FROM "参加" "参加表2"
    WHERE "参加表2"."职工号" = '1001'
);
```

如果要求“其他职工”，再加：

```
AND "参加表1"."职工号" <> '1001'
```

7. 查询每个社团有多少人参加：

```
SELECT "社团表"."社团编号", "社团表"."社团名称",
       (SELECT COUNT(DISTINCT "参加表"."职工号")
        FROM "参加" "参加表"
        WHERE "参加表"."社团编号" = "社团表"."社团编号") AS "参加人数"
FROM "社团" "社团表";
```

## 第三题：函数依赖、码与范式

已知：

```
全局关系(学号, 学生姓名, 课程号, 教师姓名, 办公室名, 成绩)
```

### 解题讲解

这一题考函数依赖、候选键、范式判断和模式分解。做这种题时，顺序一般是：先根据语义写出函数依赖，再求候选键，然后判断范式，最后按依赖把大关系分解成更合理的小关系。

第一条业务规则说：一个学生可以选多门课，一门课也可以被多个学生选。这个规则说明单独的 `学号` 不能决定 `课程号`，单独的 `课程号` 也不能决定 `学号`。学生和课程之间是多对多关系。

第二条业务规则说：一个学生在一门课上只有一个成绩。因此只要给定 `(学号, 课程号)`，成绩就唯一确定，所以有 `(学号, 课程号) -> 成绩`。

题目虽然没有单独明说“学号决定学生姓名”，但在数据库设计题中，学号通常是学生的标识，因此可写 `学号 -> 学生姓名`。如果考试特别严格，也可以说明这是根据实体标识语义得到的依赖。

第三条业务规则说：一门课程只由一个教师授课，所以 `课程号 -> 教师姓名`。一个教师只有一间办公室，所以 `教师姓名 -> 办公室名`。办公室不重复分配，说明一间办公室也只对应一个教师，所以 `办公室名 -> 教师姓名`。

候选键用属性闭包判断。先取 `(学号, 课程号)`：由 `学号` 得到学生姓名，由 `课程号` 得到教师姓名，再由教师姓名得到办公室名，由 `(学号, 课程号)` 得到成绩，所以闭包包含全部属性。因此 `(学号, 课程号)` 是超键。再看它的真子集：只有 `学号` 不能决定课程、教师、成绩；只有 `课程号` 不能决定学生和成绩。所以 `(学号, 课程号)` 是候选键。

范式判断的关键是候选键是组合键。非主属性 `学生姓名` 只依赖候选键的一部分 `学号`，非主属性 `教师姓名` 和 `办公室名` 只依赖候选键的一部分 `课程号`，这些都是部分函数依赖。因此该关系不满足第二范式，只能说最高满足第一范式。

分解时要把不同语义拆开：学生基本信息放入 `学生`，选课成绩放入 `选课`，课程与教师的对应放入 `课程`，教师与办公室的对应放入 `教师办公室`。这样每个关系表达一个比较单一的事实，既能保持原有函数依赖，也能通过共同属性自然连接恢复原关系。

业务规则推出的主要非平凡函数依赖为：

```
学号 -> 学生姓名
(学号, 课程号) -> 成绩
课程号 -> 教师姓名
教师姓名 -> 办公室名
办公室名 -> 教师姓名
```

由传递律还可推出：

```
课程号 -> 办公室名
```

候选键：

```
(学号, 课程号)
```

理由：`学号` 可确定学生姓名，`课程号` 可确定授课教师和办公室，`(学号, 课程号)` 可确定成绩，因此 `(学号, 课程号)+` 能包含全部属性；且任一真子集都不能确定全部属性。

最高范式：

```
第一范式
```

理由：各属性是不可再分的数据项，所以满足第一范式。但候选键是组合属性 `(学号, 课程号)`，而：

```
学号 -> 学生姓名
课程号 -> 教师姓名
课程号 -> 办公室名
```

这些都是非主属性对候选键真子集的部分函数依赖，因此不满足第二范式，也就不满足第三范式。

分解为第三范式关系模式：

```
学生(学号, 学生姓名)
选课(学号, 课程号, 成绩)
课程(课程号, 教师姓名)
教师办公室(教师姓名, 办公室名)
```

主键可取：

```
学生: 学号
选课: (学号, 课程号)
课程: 课程号
教师办公室: 教师姓名
```

其中 `教师办公室` 中 `办公室名` 也可作为候选键，因为办公室不重复分配。

该分解保持依赖：

```
学号 -> 学生姓名
(学号, 课程号) -> 成绩
课程号 -> 教师姓名
教师姓名 -> 办公室名
办公室名 -> 教师姓名
```

并且 `选课` 含有原关系候选键 `(学号, 课程号)`，可保证无损连接。

## 第四题：实体-联系图与关系模型

### 解题讲解

这一题考实体-联系模型，以及实体-联系图向关系模型的转换。做题时先找实体，再找实体之间的联系和联系类型，最后把它们转换成关系模式。

题目明确要管理职工、部门、商品和供应商，所以这四个就是实体。每个实体至少补充一个编号属性和一个名称属性，例如部门号、部门名，商品号、商品名。编号通常作为主键，用来唯一标识一个实体。

职工和部门之间是“一对多”：一个职工只能属于一个部门，一个部门可以有多个职工。转换成关系模型时，一对多联系通常把“一端”的主键放到“多端”作为外键，所以在 `职工` 中加入 `部门号`。

部门和商品之间也是“一对多”：一个部门可售卖多种商品，一种商品只在一个部门售卖。因此把 `部门号` 放入 `商品` 作为外键。题目要求记录销售数量，因为一种商品只对应一个销售部门，所以可以把 `销售数量` 放在 `商品` 关系中。

供应商和商品之间是“多对多”：一个供应商供应多种商品，一种商品也可由多个供应商供应。多对多联系不能简单地把某一方主键放到另一方，否则无法表示多个对应关系；必须单独转换为一个关系 `供应`。`供应` 的主键通常由 `供应商号` 和 `商品号` 共同组成，同时这两个属性也分别是外键。题目要求记录供应数量，所以把 `供应数量` 作为这个联系关系的属性。

最后写关系模式时，主键用直线标出，外键用波浪线标出。像 `供应` 中的 `供应商号` 和 `商品号`，既共同构成主键，又分别参照其他关系，所以既是主键属性，也是外键属性。

1. 实体-联系图可表示为：

```
实体：
部门(部门号, 部门名)
职工(职工号, 姓名, 部门号)
商品(商品号, 商品名, 销售数量, 部门号)
供应商(供应商号, 供应商名)

联系：
部门 -- 工作于 -- 职工        一对多
部门 -- 售卖 -- 商品          一对多
供应商 -- 供应 -- 商品        多对多，联系属性为供应数量
```

说明：

- 部门与职工是一对多，外键放在职工端。
- 部门与商品是一对多，外键放在商品端；销售数量可作为商品或销售联系的属性。由于一种商品只在一个部门售卖，这里并入商品关系。
- 供应商与商品是多对多，必须转换为独立关系 `供应`，并带有联系属性 `供应数量`。

2. 转换后的关系模型：

```
部门(部门号, 部门名)
职工(职工号, 姓名, 部门号)
商品(商品号, 商品名, 销售数量, 部门号)
供应商(供应商号, 供应商名)
供应(供应商号, 商品号, 供应数量)
```

3. 关系模式。下面用 `___...___` 表示主键，用 `﹏...﹏` 表示外键；既是主键又是外键时两者同时标出。

```
部门(___部门号___, 部门名)

职工(___职工号___, 姓名, ﹏部门号﹏)

商品(___商品号___, 商品名, 销售数量, ﹏部门号﹏)

供应商(___供应商号___, 供应商名)

供应(___﹏供应商号﹏___, ___﹏商品号﹏___, 供应数量)
```

外键参照：

```
职工.部门号 -> 部门.部门号
商品.部门号 -> 部门.部门号
供应.供应商号 -> 供应商.供应商号
供应.商品号 -> 商品.商品号
```

## 第五题：安全性控制语句

已知：

```
职工表(职工号, ..., 工资)
部门表(部门号, ...)
```

### 解题讲解

这一题考数据库安全性中的自主存取控制。核心语句是授权语句，基本格式是：把某种权限授予某个用户，并指定权限作用在哪个表、视图或属性列上。

第 1 小问授予查询权限，使用 `GRANT SELECT`。因为有两张表，所以分别对 `职工表` 和 `部门表` 授权给 `王明`。

第 2 小问授予插入和删除权限，可以把两个权限写在同一条授权语句中，即 `GRANT INSERT, DELETE`。同样要分别对两张表授权给 `刘星`。

第 3 小问要求员工只能查询自己的记录。只靠给基本表授权很难限制“只能看自己的行”，所以要用视图机制：先定义一个只包含某员工本人记录的视图，再只把该视图的查询权限授给该员工。这样员工访问的是视图，不是完整的基本表。题中用 `王明员工视图` 举例，其他员工照此建立各自视图。

第 4 小问要求 `刘星` 可以查询整张职工表，但只能修改工资字段。授权语句支持列级权限，因此写 `UPDATE("工资")`，表示更新权限只作用在 `工资` 这一列上，而不是整张表所有列。

第 5 小问要求授予全部权限，并允许继续转授权。全部权限用 `ALL PRIVILEGES`，允许把权限再授给其他用户用 `WITH GRANT OPTION`。因为对象有两张表，所以两张表各写一条授权语句。

1. 授予用户 `王明` 对两张表的查询权限：

```
GRANT SELECT
ON TABLE "职工表"
TO "王明";

GRANT SELECT
ON TABLE "部门表"
TO "王明";
```

2. 授予用户 `刘星` 对两张表的插入和删除权限：

```
GRANT INSERT, DELETE
ON TABLE "职工表"
TO "刘星";

GRANT INSERT, DELETE
ON TABLE "部门表"
TO "刘星";
```

3. 员工只能查询自己的记录。按图谱中的视图机制：为不同员工定义只含本人记录的视图，并只把该视图权限授予对应员工。

```
CREATE VIEW "王明员工视图"
AS
SELECT *
FROM "职工表"
WHERE "职工号" = '王明';

GRANT SELECT
ON TABLE "王明员工视图"
TO "王明";
```

对其他员工同理，把中文视图名、`职工号` 条件值和授权用户改成对应员工即可。若系统用户名与 `职工号` 值一致，也可把视图条件写成：

```
WHERE "职工号" = USER
```

要点：不要把 `职工表` 基本表的查询权限直接授予普通员工，只授予对应视图的查询权限。

4. 授予用户 `刘星` 查询 `职工表`，以及只修改 `工资` 字段的权限：

```
GRANT SELECT, UPDATE("工资")
ON TABLE "职工表"
TO "刘星";
```

5. 授予用户 `王明` 对两张表的全部权限，并允许继续转授权：

```
GRANT ALL PRIVILEGES
ON TABLE "职工表"
TO "王明"
WITH GRANT OPTION;

GRANT ALL PRIVILEGES
ON TABLE "部门表"
TO "王明"
WITH GRANT OPTION;
```
