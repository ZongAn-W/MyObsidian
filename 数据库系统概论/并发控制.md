## 7. 活锁和死锁

### 7.1 活锁

活锁指某个事务一直等待某个数据对象的锁，但系统不断把锁授予其他事务，使该事务长期得不到执行机会。

例：

```text
T1 等待锁 R
T2 申请锁 R，系统给 T2
T3 申请锁 R，系统又给 T3
...
T1 一直等待
```

避免活锁的常用方法：

```text
先来先服务
```

即按申请锁的先后顺序排队。

### 7.2 死锁

死锁指两个或多个事务各自持有对方需要的锁，互相等待，导致都无法继续执行。

例：

```text
T1 已锁 R1，等待 R2
T2 已锁 R2，等待 R1
```

本质：

```text
循环等待
```

### 7.3 活锁与死锁区别

| 项目 | 活锁 | 死锁 |
|---|---|---|
| 表现 | 某事务不断被插队，长期等待 | 多个事务互相等待 |
| 是否循环等待 | 不一定 | 是 |
| 常用处理 | 先来先服务 | 预防、检测、解除 |
| 是否必须撤销事务 | 通常不必 | 常需撤销某个事务 |

## 8. 死锁处理

死锁处理有两类方法：

1. 预防死锁；
2. 诊断并解除死锁。

### 8.1 死锁预防

目标：破坏死锁产生的条件。

常见方法：

#### 一次封锁法

一次封锁法要求事务一次性申请它要使用的全部数据对象的锁。  
如果不能全部加锁，就不执行。

优点：

- 可以防止死锁。

缺点：

- 扩大封锁范围；
- 降低并发度；
- 很难事先准确知道事务需要哪些锁。

#### 顺序封锁法

顺序封锁法要求对数据对象规定统一的封锁顺序，所有事务按同一顺序申请锁。

优点：

- 可防止循环等待；
- 能预防死锁。

缺点：

- 维护统一顺序困难；
- 数据对象动态变化时实现成本高；
- 可能降低并发度。

### 8.2 死锁诊断

常用方法：

1. 超时法；
2. 事务等待图法。

#### 超时法

如果事务等待时间超过规定时限，就认为可能发生死锁。

优点：

- 实现简单。

缺点：

- 可能误判；
- 超时时间过长会发现太晚，过短会误撤销事务。

#### 事务等待图法

事务等待图是有向图：

- 结点表示事务；
- 边表示等待关系；
- 若 `T1` 等待 `T2` 持有的锁，则有边 `T1 -> T2`。

判断规则：

```text
等待图有环 -> 存在死锁
等待图无环 -> 没有死锁
```

### 8.3 死锁解除

检测到死锁后，系统通常选择一个代价较小的事务撤销，释放其持有的锁，使其他事务继续执行。

选择牺牲事务时可考虑：

- 已执行时间；
- 已修改数据量；
- 持有锁数量；
- 回滚代价；
- 事务优先级。

## 9. 并发调度的可串行性

### 9.1 调度

调度是多个事务操作的执行序列。

串行调度：事务一个接一个完整执行。  
并发调度：不同事务的操作交叉执行。

### 9.2 可串行化调度

一个并发调度，如果其执行结果与某个串行调度的结果相同，则称为可串行化调度。

可串行化是并发事务正确调度的准则。

记忆：

```text
并发执行可以交叉，但结果要像某种串行执行。
```

### 9.3 冲突操作

不同事务对同一数据对象的读写操作，如果至少有一个是写操作，则称为冲突操作。

冲突操作包括：

```text
Ri(x) 与 Wj(x)
Wi(x) 与 Rj(x)
Wi(x) 与 Wj(x)
```

其中 `i != j`。

非冲突操作：

```text
Ri(x) 与 Rj(x)
不同数据对象上的操作
同一事务内部操作
```

### 9.4 冲突可串行化

如果一个调度可以通过交换相邻的非冲突操作，变换成某个串行调度，则称为冲突可串行化调度。

重要结论：

```text
冲突可串行化调度一定是可串行化调度。
```

但反过来不一定成立：

```text
可串行化调度不一定冲突可串行化。
```

### 9.5 判断思路

给定调度判断是否冲突可串行化：

1. 找出不同事务对同一数据对象的冲突操作。
2. 根据冲突操作的先后顺序建立事务优先关系。
3. 如果优先关系中无循环，通常可变换为某个串行调度。
4. 如果存在循环，则不是冲突可串行化。

## 10. 两段锁协议

### 10.1 定义

两段锁协议要求所有事务分两个阶段进行加锁和解锁：

1. 扩展阶段：事务可以申请锁，但不能释放锁。
2. 收缩阶段：事务可以释放锁，但不能再申请新锁。

也就是说：

```text
一旦释放了任何锁，就不能再申请新的锁。
```

### 10.2 作用

重要结论：

```text
若所有事务都遵守两段锁协议，则这些事务的任何并发调度都是可串行化的。
```

因此，两段锁协议是保证并发调度可串行化的一种充分条件。

### 10.3 注意点

两段锁协议不是可串行化的必要条件。

也就是说：

- 遵守两段锁协议，一定可串行化；
- 可串行化调度，不一定由两段锁协议产生。

两段锁协议也不能保证不发生死锁。

例：

```text
T1 已加 S(B)，再申请 X(A)
T2 已加 S(A)，再申请 X(B)
```

两个事务互相等待，仍可能死锁。

### 10.4 两段锁协议与一次封锁法

| 项目 | 两段锁协议 | 一次封锁法 |
|---|---|---|
| 加锁方式 | 分阶段逐步加锁 | 一开始一次申请全部锁 |
| 是否保证可串行化 | 是 | 是 |
| 是否防止死锁 | 不一定 | 是 |
| 并发度 | 较高 | 较低 |
| 要求是否提前知道所有数据对象 | 不一定 | 必须 |

## 11. 封锁的粒度

### 11.1 封锁粒度定义

封锁对象的大小称为封锁粒度。

封锁对象可以是：

- 属性值；
- 元组；
- 关系；
- 索引项；
- 整个数据库；
- 数据页；
- 物理记录。

### 11.2 粒度大小的影响

封锁粒度越大：

- 系统开销越小；
- 并发度越低；
- 冲突可能越多。

封锁粒度越小：

- 并发度越高；
- 系统开销越大；
- 加锁和解锁管理更复杂。

记忆：

```text
大粒度：省事但并发差
小粒度：并发好但开销大
```

### 11.3 多粒度封锁

多粒度封锁允许系统支持不同大小的封锁对象。

多粒度树示例：

```text
数据库
  -> 关系
      -> 元组
```

对某个结点加锁，意味着对其所有后裔结点也施加相应控制。

### 11.4 显式封锁与隐式封锁

- 显式封锁：事务直接对某个数据对象加锁。
- 隐式封锁：由于对上级结点加锁，下级结点自动受到封锁约束。

问题：

如果只检查显式锁，可能漏掉上级结点带来的隐式锁。  
为降低检查成本，引入意向锁。

## 12. 意向锁

### 12.1 意向锁的作用

意向锁表示事务打算在某个结点的后裔结点上加锁。

作用：

```text
提前在上级结点留下标记，避免系统逐个检查所有下级结点。
```

例：

如果事务要对关系 `R1` 中某个元组加 S 锁，需要先对数据库和关系 `R1` 加意向锁。

### 12.2 常见意向锁类型

常见三种：

1. `IS`：意向共享锁。
2. `IX`：意向排他锁。
3. `SIX`：共享意向排他锁。

### 12.3 IS 锁

`IS` 表示事务准备在后裔结点上加 S 锁。

例：

```text
要对某个元组加 S 锁
先对其所在关系和数据库加 IS 锁
```

### 12.4 IX 锁

`IX` 表示事务准备在后裔结点上加 X 锁。

例：

```text
要修改某个元组
先对其所在关系和数据库加 IX 锁
再对元组加 X 锁
```

### 12.5 SIX 锁

`SIX` 等价于：

```text
S + IX
```

含义：

- 对当前结点加 S 锁；
- 同时打算对部分后裔结点加 X 锁。

例：

事务要读整个表，同时修改其中若干元组，可对表加 `SIX` 锁。

### 12.6 意向锁的意义

意向锁提高了多粒度封锁的效率。

例如，事务要对整个表加 X 锁时，系统只需检查表及其上级结点是否有冲突的意向锁，而不必扫描表中每个元组是否已加锁。

## 13. 其他并发控制机制

除封锁外，还常见：

- 时间戳方法；
- 乐观并发控制；
- 多版本并发控制，简称 MVCC。

### 13.1 时间戳方法

给每个事务分配唯一时间戳，按时间戳顺序解决冲突。

特点：

- 不依赖传统封锁等待；
- 冲突时可能回滚较晚或较早的事务；
- 通过时间顺序保证可串行化。

### 13.2 乐观并发控制

乐观并发控制假设冲突较少，事务执行时不加锁，提交前检查是否发生冲突。

基本阶段：

1. 读阶段；
2. 验证阶段；
3. 写阶段。

适合：

- 读多写少；
- 冲突较少的环境。

### 13.3 多版本并发控制

MVCC 通过维护数据对象的多个版本提高并发度。

基本思想：

- 写操作创建新版本；
- 读操作可读取适合自己时间点的旧版本；
- 读写冲突减少。

优点：

- 提高读写并发；
- 读事务通常不阻塞写事务；
- 写事务通常不阻塞读事务。

## 14. 高频考点

### 14.1 简答题

**并发操作可能带来哪些数据不一致？**

答：

```text
丢失修改、脏读、不可重复读、幻读。
```

**X 锁和 S 锁的区别是什么？**

答：

```text
X 锁用于写，排他；S 锁用于读，可共享。
S 锁之间相容，X 锁与任何锁都不相容。
```

**三级封锁协议分别能解决什么问题？**

答：

```text
一级防丢失修改；
二级防丢失修改和脏读；
三级防丢失修改、脏读和不可重复读。
```

**两段锁协议的核心要求是什么？**

答：

```text
先加锁后解锁；释放任一锁后不能再申请新锁。
```

**两段锁协议是否能避免死锁？**

答：不能。两段锁协议能保证可串行化，但仍可能发生死锁。

### 14.2 判断题

常见陷阱：

- “S 锁和 S 锁不相容。”  
  错。多个事务可以同时持有 S 锁。

- “X 锁只禁止其他事务写，不禁止读。”  
  错。X 锁禁止其他事务加任何锁，读写都受限。

- “两段锁协议是可串行化的必要条件。”  
  错。它是充分条件，不是必要条件。

- “可串行化调度一定是冲突可串行化调度。”  
  错。冲突可串行化一定可串行化，反之不一定。

- “封锁粒度越小越好。”  
  错。粒度越小并发度高，但系统开销也大。

### 14.3 分析题

常见题型：

1. 根据两个事务的交叉执行，判断发生了丢失修改、脏读、不可重复读还是幻读。
2. 给出封锁序列，判断是否遵守两段锁协议。
3. 给出等待关系，画事务等待图并判断是否死锁。
4. 给出调度，判断是否冲突可串行化。
5. 根据隔离级别判断能否避免某种并发异常。

解题套路：

```text
判断并发异常：
看是否读到未提交 -> 脏读
看是否同一行两次值不同 -> 不可重复读
看是否记录集合变了 -> 幻读
看是否写覆盖写 -> 丢失修改

判断死锁：
画等待图，有环就是死锁

判断两段锁：
找第一次 UNLOCK
如果之后又 LOCK，则违反两段锁

判断冲突：
同一数据对象、不同事务、至少一个写
```
