# 第十章 关系查询处理和查询优化：重要知识点总结

> 来源：《数据库系统概论（第6版）》第10章。重点掌握查询处理步骤、选择和连接操作的实现算法、查询优化的基本思想、代数优化规则、物理优化策略和代价估算。

## 1. 本章学习主线

关系数据库的一个重要特点是：用户只需要用 SQL 说明“要什么”，不用说明“怎么取”。  
把 SQL 转换为高效执行计划，就是 DBMS 查询处理和查询优化的核心任务。

本章主线：

```text
SQL 查询语句
    -> 查询分析
    -> 查询检查
    -> 查询优化
        -> 代数优化：改写关系代数表达式
        -> 物理优化：选择具体存取路径和算法
    -> 查询执行
```

考试中常考：

- 查询处理分哪几个阶段？
- 查询优化为什么重要？
- 选择、连接操作有哪些实现算法？
- 代数优化的基本规则是什么？
- 物理优化如何选择索引、排序、连接算法？
- 如何从 I/O 块数角度估算查询代价？

## 2. 查询处理的基本步骤

关系 DBMS 对查询语句的处理通常包括 4 个阶段：

1. 查询分析
2. 查询检查
3. 查询优化
4. 查询执行

### 2.1 查询分析

查询分析主要进行：

- 词法分析：识别 SQL 关键字、关系名、属性名、常量、运算符等。
- 语法分析：检查 SQL 语句是否符合语法规则。

如果查询语句有语法错误，系统会拒绝继续处理。

例：

```sql
SELECT Sname
FROM Student
WHERE Sno = '20180001';
```

系统要识别：

- `SELECT`、`FROM`、`WHERE` 是关键字；
- `Student` 是关系名；
- `Sname`、`Sno` 是属性名；
- `'20180001'` 是常量。

### 2.2 查询检查

查询检查是在语义层面判断查询是否合法，主要包括：

- 关系名、属性名是否存在；
- 属性是否属于对应关系；
- 用户是否有访问权限；
- 视图是否需要转换为基本表操作；
- 完整性约束和安全性约束是否允许该查询；
- 将外部名字转换成内部表示。

检查通过后，SQL 查询会转换为内部表示，通常可表示为语法树或关系代数表达式。

### 2.3 查询优化

查询优化是从多个等价执行方案中选择较高效方案的过程。

优化可分为：

- **代数优化**：通过关系代数等价变换，改变操作顺序和组合方式。
- **物理优化**：选择存取路径和底层操作算法，如是否使用索引、采用哪种连接算法。

常见优化依据：

- 基于规则：使用经验性优化规则。
- 基于代价：估计各执行计划代价，选择代价较小的计划。
- 基于语义：利用约束、完整性、语义信息进行优化。

### 2.4 查询执行

查询优化器生成查询执行计划后，由代码生成器生成执行代码，系统按计划访问数据、执行运算并返回结果。

## 3. 查询优化为什么重要

同一个 SQL 查询可能有很多等价执行方式，但代价可能差别极大。

例如：

```sql
SELECT Student.Sname
FROM Student, SC
WHERE Student.Sno = SC.Sno
  AND SC.Cno = '81003';
```

可以有多种执行思路：

1. 先做 `Student × SC` 或连接，再选择 `Cno='81003'`；
2. 先从 `SC` 中选择 `Cno='81003'` 的少量记录，再与 `Student` 连接；
3. 如果 `SC.Cno` 或 `Student.Sno` 上有索引，先通过索引定位记录，再连接。

通常第 2、3 种会比先做大规模连接或笛卡尔积高效得多。

高频结论：

- **先选择、后连接**通常能减少中间结果。
- **尽早投影**可以减少中间结果宽度。
- **利用索引**可以减少磁盘 I/O。
- **连接顺序**会显著影响执行代价。

## 4. 实现查询操作的算法

### 4.1 选择操作的实现

选择操作形如：

```sql
SELECT *
FROM Student
WHERE 条件;
```

常见实现方式：

1. 全表扫描
2. 索引扫描

### 4.2 全表扫描

全表扫描也叫顺序扫描。

基本思想：

- 逐块读入关系中的所有数据块；
- 对每个元组检查选择条件；
- 满足条件的元组作为结果输出。

特点：

- 实现简单；
- 不依赖索引；
- 对小表可接受；
- 对大表且选择率低时效率较差。

适合场景：

- 小表；
- 查询条件无可用索引；
- 查询结果占全表比例很高；
- `OR` 条件复杂且无法有效利用索引。

### 4.3 索引扫描

索引扫描通过索引先找到满足条件元组的指针，再访问数据表。

适合场景：

- 选择条件属性上有索引；
- 选择率较低；
- 查询条件是主码等值查询；
- 范围查询可使用 B+ 树索引。

例：

```sql
SELECT *
FROM Student
WHERE Sno = '20180003';
```

如果 `Sno` 上有索引，可以直接通过索引定位对应元组。

范围查询：

```sql
SELECT *
FROM Student
WHERE Sbirthdate >= '2000-01-01';
```

如果 `Sbirthdate` 上有 B+ 树索引，可先定位范围起点，再沿叶子结点顺序扫描。

复合条件：

```sql
WHERE Smajor = '计算机科学与技术'
  AND Sbirthdate >= '2000-01-01'
```

可能策略：

- 两个属性都有索引：分别得到候选元组指针，再求交集；
- 只有一个属性有索引：先用索引定位候选元组，再检查另一个条件；
- 没有可用索引：全表扫描。

## 5. 连接操作的实现

连接是查询处理中最常见、最耗时的操作之一。

常见连接算法：

1. 嵌套循环连接
2. 排序-合并连接
3. 索引连接
4. 哈希连接

### 5.1 嵌套循环连接

嵌套循环连接的基本思想：

- 外层循环读取外表的元组；
- 内层循环扫描内表的元组；
- 检查连接条件；
- 满足条件则拼接输出。

伪过程：

```text
for each tuple r in R:
    for each tuple s in S:
        if r.A = s.B:
            output r join s
```

特点：

- 算法简单；
- 不要求表有序；
- 不要求索引；
- 数据量大时可能代价很高。

优化方式：

- 以较小的表作为外表；
- 尽可能使用内存缓冲区多读外表块；
- 若内表连接属性有索引，可转为索引连接。

### 5.2 排序-合并连接

排序-合并连接适合等值连接，尤其适合两个表已经按连接属性排序的情况。

基本步骤：

1. 如果两个表没有按连接属性排序，先排序。
2. 同时顺序扫描两个有序表。
3. 找到连接属性相等的元组组。
4. 将相等组中的元组配对输出。

特点：

- 适合等值连接或自然连接；
- 如果表已有序，代价较低；
- 每个表通常只需顺序扫描；
- 若需要额外排序，需加上排序代价。

### 5.3 索引连接

索引连接适用于一个表的连接属性上有索引的情况。

基本思想：

- 扫描外表；
- 对外表每个连接值，通过内表索引查找匹配元组；
- 拼接输出。

适合场景：

- 内表连接属性有索引；
- 外表较小；
- 连接选择率较低。

例：

```sql
SELECT *
FROM Student, SC
WHERE Student.Sno = SC.Sno;
```

如果 `SC.Sno` 上有索引，可扫描 `Student`，对每个学生号到 `SC` 索引中查找对应选课记录。

### 5.4 哈希连接

哈希连接适合等值连接。

基本思想：

1. 划分阶段：对较小表按连接属性哈希，将元组分配到哈希桶中。
2. 探测阶段：扫描另一个表，用同一哈希函数定位桶，查找匹配元组。

特点：

- 适合大表等值连接；
- 不适合非等值连接；
- 若较小表能放入内存，效率更高；
- 若哈希桶溢出，需要额外处理。

## 6. 查询优化的分类

### 6.1 代数优化

代数优化也叫逻辑优化。

核心：利用关系代数等价变换，改变查询表达式的结构，使中间结果尽量小。

常见目标：

- 选择下推；
- 投影下推；
- 连接代替笛卡尔积后选择；
- 调整连接顺序；
- 合并连续选择和投影。

### 6.2 物理优化

物理优化关注“如何执行”。

核心：为关系代数操作选择具体算法和存取路径。

常见选择：

- 是否使用索引；
- 使用全表扫描还是索引扫描；
- 连接用嵌套循环、排序-合并、索引连接还是哈希连接；
- 是否排序；
- 哪个表作为外表；
- 中间结果是否物化。

### 6.3 两者区别

| 类型 | 又称 | 优化对象 | 关注点 | 例子 |
|---|---|---|---|---|
| 代数优化 | 逻辑优化 | 关系代数表达式 | 操作顺序和组合 | 选择下推、投影下推 |
| 物理优化 | 存取路径优化 | 执行计划 | 具体算法和访问方式 | 使用索引、选择哈希连接 |

易混点：

- 代数优化不关心底层索引怎么用。
- 物理优化不一定改变查询逻辑结构，但会改变执行方式。
- 实际 DBMS 通常把两者结合使用。

## 7. 关系代数等价变换规则

关系代数表达式等价：两个表达式对任意合法数据库实例得到相同结果。

常考规则包括：

### 7.1 连接与笛卡尔积的交换律

```text
E1 × E2 = E2 × E1
E1 ⋈ E2 = E2 ⋈ E1
```

含义：两个关系的连接顺序可以交换。

### 7.2 连接与笛卡尔积的结合律

```text
(E1 × E2) × E3 = E1 × (E2 × E3)
(E1 ⋈ E2) ⋈ E3 = E1 ⋈ (E2 ⋈ E3)
```

含义：多个关系连接时，可重新安排连接顺序。

### 7.3 选择串接律

```text
σF1(σF2(E)) = σF1 ∧ F2(E)
```

含义：多个选择条件可以合并，也可以拆开分别下推。

### 7.4 投影串接律

```text
πA(πB(E)) = πA(E)   （A 是 B 的属性子集）
```

含义：连续投影可以合并，只保留最终需要属性。

### 7.5 选择与投影交换

如果选择条件只涉及投影保留的属性：

```text
σF(πA(E)) = πA(σF(E))
```

含义：可先选择再投影，减少元组数。

### 7.6 选择与连接交换

如果选择条件只涉及 `E1` 的属性：

```text
σF(E1 ⋈ E2) = σF(E1) ⋈ E2
```

如果条件可拆分：

```text
σF1 ∧ F2(E1 ⋈ E2) = σF1(E1) ⋈ σF2(E2)
```

含义：选择应尽可能下推到基表。

### 7.7 选择与集合运算分配

对并、差等集合运算，若条件适用，可将选择分配到两个子表达式：

```text
σF(E1 ∪ E2) = σF(E1) ∪ σF(E2)
σF(E1 - E2) = σF(E1) - σF(E2)
```

### 7.8 投影与连接、集合运算

投影也可以在满足属性需求的前提下下推：

```text
πA(E1 ⋈ E2)
```

可改写为先对 `E1`、`E2` 投影保留必要属性，再连接。

注意：

- 不能过早丢掉后续连接、选择、输出所需属性。
- 投影下推的目标是减少中间结果宽度。

## 8. 启发式代数优化规则

常见启发式规则：

1. **选择运算尽早执行**  
   选择通常大幅减少元组数，是最重要的优化规则。

2. **投影运算尽早执行**  
   投影减少属性列数，降低中间结果存储和传输代价。

3. **选择与投影同时下推**  
   对同一关系的选择和投影可以尽量在一次扫描中完成。

4. **把选择与笛卡尔积合并为连接**  
   形如：

   ```text
   σR.A=S.B(R × S)
   ```

   应改写为：

   ```text
   R ⋈R.A=S.B S
   ```

5. **优先连接能显著缩小结果的关系**

   例如先连接有高选择性条件的关系，避免产生巨大中间结果。

6. **重复出现的公共子表达式可先计算一次**

   如果公共子表达式结果不大，可物化后重复使用。

## 9. 物理优化策略

### 9.1 选择操作的启发式规则

常见策略：

- 小关系：可直接全表扫描。
- 主码等值查询：优先使用主码索引。
- 非主属性等值查询：若有索引且选择率低，可使用索引。
- 范围查询：若有 B+ 树索引且选择率低，可使用索引。
- `AND` 条件：优先考虑组合索引；也可分别用单属性索引后求交。
- `OR` 条件：很多情况下全表扫描更合适，除非各分支都能高效利用索引。

### 9.2 连接操作的启发式规则

常见策略：

- 若两个表已经按连接属性排序，可选排序-合并连接。
- 若一个表连接属性有索引，可选索引连接。
- 若一个表较小，可选哈希连接。
- 若没有索引、无序且数据量不大，可用嵌套循环连接。
- 嵌套循环连接中，通常让较小表作为外表。

### 9.3 基于代价估算的优化

基于代价的优化器会估算多个执行计划的代价，选择代价较小者。

常见代价组成：

```text
总代价 = I/O 代价 + CPU 代价 + 内存代价 + 通信代价
```

集中式数据库中，通常最关注磁盘 I/O 代价，因为磁盘读写比内存操作慢得多。

分布式数据库还要考虑通信代价。

## 10. 统计信息

代价估算依赖数据库统计信息。

常用统计信息包括：

### 10.1 对基本表

- 元组总数 `N`；
- 元组长度 `l`；
- 占用块数 `B`。

### 10.2 对属性列

- 不同值个数；
- 最大值、最小值；
- 值分布是否均匀；
- 是否建立索引；
- 选择率。

### 10.3 对索引

- 索引类型，如 B+ 树索引；
- 索引层数 `L`；
- 不同索引值个数；
- 索引选择基数；
- 叶结点块数。

记忆：

```text
没有统计信息，优化器只能猜；
统计信息越准确，代价估算越可靠。
```

## 11. 选择率与代价估算

### 11.1 选择率

选择率表示满足选择条件的元组占总元组的比例。

```text
选择率 = 满足条件的元组数 / 总元组数
```

选择率低：结果少，适合使用索引。  
选择率高：结果多，全表扫描可能更划算。

### 11.2 全表扫描代价

若关系占用 `B` 个数据块：

```text
全表扫描代价约为 B
```

若查询条件为主码等值查询，平均可能读到一半表块：

```text
平均代价约为 B / 2
```

### 11.3 索引扫描代价

索引扫描代价通常包括：

```text
索引访问代价 + 数据块访问代价
```

对于 B+ 树索引：

```text
代价约为 索引层数 L + 访问满足条件元组所在数据块的代价
```

如果选择率低，索引扫描常优于全表扫描。  
如果选择率高，反复通过索引访问数据块可能比顺序扫描更慢。

### 11.4 连接代价

连接代价与以下因素有关：

- 两表占用块数；
- 内存缓冲区块数；
- 连接选择率；
- 是否有索引；
- 是否已有序；
- 中间结果是否需要写回磁盘。

常见考试思路：

1. 先判断能否利用索引。
2. 判断是否已有序。
3. 判断哪个表更小。
4. 估算读写块数。
5. 选择代价较低的连接算法。

## 12. 查询计划的执行

查询优化后，DBMS 会形成查询计划。

查询计划执行方式常见两类：

### 12.1 自顶向下执行

特点：

- 根结点向子结点发出获取元组请求；
- 子结点按需返回结果；
- 类似需求驱动；
- 不一定需要完全物化所有中间结果。

### 12.2 自底向上执行

特点：

- 从叶结点开始计算；
- 逐步生成中间结果；
- 中间结果可能放入缓冲区或临时文件；
- 父结点再读取子结点结果继续执行。

易混点：

- 自顶向下更像“需要结果时再向下取”。
- 自底向上更像“先算子查询结果，再向上组合”。

## 13. 高频考点

### 13.1 简答题

**查询处理分哪几个阶段？**

答：

```text
查询分析、查询检查、查询优化、查询执行。
```

**查询优化分哪两类？**

答：

```text
代数优化和物理优化。
```

**为什么选择运算要尽早执行？**

答：选择能减少元组数，降低后续连接、排序、投影等操作的中间结果规模，从而减少 I/O 和 CPU 代价。

**为什么投影要尽早执行？**

答：投影能减少属性列数，降低中间结果宽度，减少存储、传输和比较成本。

### 13.2 计算题

可能要求：

- 根据关系块数估算全表扫描代价；
- 比较全表扫描和索引扫描；
- 比较嵌套循环连接、排序-合并连接、索引连接的 I/O 代价；
- 判断给定查询应选哪种执行策略。

解题套路：

```text
先看选择条件：
    是否主码等值？
    是否有索引？
    选择率高不高？

再看连接：
    是否等值连接？
    是否有序？
    是否有索引？
    哪个表小？

最后估算：
    读多少块？
    写多少中间结果？
    是否需要排序？
```

### 13.3 关系代数优化题

常见要求：

- 将 SQL 转换为语法树或关系代数表达式；
- 用等价变换优化；
- 画出优化后的语法树；
- 说明选择下推、投影下推、连接替换笛卡尔积。

常用答题套路：

```text
1. 把 FROM 中的多个表先写成笛卡尔积。
2. 把 WHERE 条件写成选择。
3. 把 SELECT 列写成投影。
4. 拆分选择条件。
5. 将只涉及单表的选择下推到对应基表。
6. 将连接条件与笛卡尔积合并为连接。
7. 投影下推，但保留连接属性和最终输出属性。
```

## 14. 易混概念辨析

| 易混点 | 正确认识 |
|---|---|
| 查询分析 vs 查询检查 | 分析偏语法，检查偏语义、权限、完整性 |
| 代数优化 vs 物理优化 | 代数优化改写表达式，物理优化选择算法和存取路径 |
| 全表扫描 vs 索引扫描 | 全表扫描顺序读全表，索引扫描先查索引再取数据 |
| 排序-合并连接 vs 哈希连接 | 前者依赖有序或需排序，后者依赖哈希且适合等值连接 |
| 索引一定更快吗 | 不一定，选择率高时全表扫描可能更快 |
| 选择下推 vs 投影下推 | 前者减少行数，后者减少列数 |
| 基于规则 vs 基于代价 | 前者用经验规则，后者估算执行代价 |

```text
查询处理四步：
分析、检查、优化、执行

优化两类：
代数优化改逻辑，物理优化选算法

选择实现：
全表扫描、索引扫描

连接实现：
嵌套循环、排序合并、索引连接、哈希连接

代数优化核心：
选择下推、投影下推、连接替换笛卡尔积、重排连接顺序

物理优化核心：
用统计信息估代价，少读磁盘块

索引使用原则：
选择率低用索引，选择率高未必用索引

连接选择：
有序用排序合并，有索引用索引连接，小表可哈希，无条件用嵌套循环
```
