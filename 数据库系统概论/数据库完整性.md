# 第五章 数据库完整性：重要知识点总结

> 来源：《数据库系统概论（第6版）》第5章。重点掌握完整性约束、实体完整性、参照完整性、用户定义完整性、命名约束、断言和触发器。

## 1. 本章学习主线

数据库完整性是指数据库中数据的正确性和相容性。

本章主要回答三个问题：

1. 什么是数据库完整性？
2. 如何用 SQL 定义和检查完整性约束？
3. 违反完整性约束时，DBMS 如何处理？

核心记忆：

```text
完整性 = 正确性 + 相容性
完整性控制 = 定义约束 + 检查约束 + 违约处理
```

## 2. 数据库完整性概述

### 2.1 完整性的含义

数据库完整性是指数据库中数据的正确性和相容性。

- 正确性：数据符合现实世界语义，能反映当前实际情况。
- 相容性：同一对象在不同关系表中的数据相同、一致。

例子：

- 学号必须唯一。
- 性别只能取“男”或“女”。
- 百分制成绩必须在 0 到 100 之间。
- 学生选修的课程必须是学校开设的课程。

### 2.2 完整性与安全性的区别

| 概念 | 防范对象 | 目标 |
| --- | --- | --- |
| 完整性 | 不正确、不合语义的数据 | 保证数据正确、有效、相容 |
| 安全性 | 非法用户和非法操作 | 防止数据泄露、篡改、破坏 |

### 2.3 DBMS 的完整性控制功能

DBMS 为维护完整性，需要提供三类功能：

1. 定义完整性约束的机制。
2. 检查完整性约束的方法。
3. 违约处理方法。

完整性约束一般作为数据库模式的一部分存入数据字典。执行 `INSERT`、`UPDATE`、`DELETE` 时，DBMS 会检查操作是否违反约束。

## 3. 实体完整性

### 3.1 实体完整性的含义

实体完整性要求基本关系的主属性不能取空值，主码必须唯一标识元组。

关系模型中，实体完整性通常通过 `PRIMARY KEY` 定义。

### 3.2 单属性主码

列级定义：

```sql
CREATE TABLE Student (
    Sno CHAR(8) PRIMARY KEY,
    Sname VARCHAR(20),
    Ssex CHAR(1)
);
```

表级定义：

```sql
CREATE TABLE Student (
    Sno CHAR(8),
    Sname VARCHAR(20),
    Ssex CHAR(1),
    PRIMARY KEY (Sno)
);
```

### 3.3 多属性主码

多属性主码只能用表级约束定义。

```sql
CREATE TABLE SC (
    Sno CHAR(8),
    Cno CHAR(5),
    Grade SMALLINT,
    Semester CHAR(5),
    Teachingclass CHAR(8),
    PRIMARY KEY (Sno, Cno)
);
```

### 3.4 实体完整性检查

DBMS 检查：

- 主码值是否为空。
- 主码值是否唯一。

若插入或修改后的主码值为空，或与已有主码重复，则操作被拒绝。

常见实现：DBMS 通常会在主码上自动建立索引，以提高唯一性检查效率。

## 4. 参照完整性

### 4.1 参照完整性的含义

参照完整性描述关系与关系之间的引用约束。

若关系 `R` 的外码 `F` 参照关系 `S` 的主码 `Ks`，则 `R` 中 `F` 的值必须满足：

- 等于 `S` 中某个元组的主码值；
- 或者取空值。

通俗理解：外码不能引用不存在的对象。

### 4.2 参照完整性的定义

```sql
FOREIGN KEY (<外码列>)
REFERENCES <被参照表>(<主码列>)
```

例子：

```sql
CREATE TABLE SC (
    Sno CHAR(8),
    Cno CHAR(5),
    Grade SMALLINT,
    Semester CHAR(5),
    Teachingclass CHAR(8),
    PRIMARY KEY (Sno, Cno),
    FOREIGN KEY (Sno) REFERENCES Student(Sno),
    FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

### 4.3 参照完整性的违约处理

当删除或修改被参照表中的元组，导致参照关系不一致时，可以采用不同策略。

常见策略：

| 策略 | 含义 |
| --- | --- |
| NO ACTION | 拒绝执行该操作 |
| CASCADE | 级联执行相关操作 |
| SET NULL | 将参照关系中的外码值设为空 |
| SET DEFAULT | 将外码值设为默认值 |

示例：

```sql
FOREIGN KEY (Sno) REFERENCES Student(Sno)
ON DELETE CASCADE
ON UPDATE CASCADE
```

含义：

- 删除 Student 中某学生时，级联删除 SC 中该学生的选课记录。
- 修改 Student 中该学生学号时，级联修改 SC 中对应学号。

另一个例子：

```sql
FOREIGN KEY (Cno) REFERENCES Course(Cno)
ON DELETE NO ACTION
ON UPDATE CASCADE
```

含义：

- 若 SC 中仍有学生选了某课程，则不允许删除 Course 中该课程。
- 若 Course 中课程号更新，则级联更新 SC 中课程号。

## 5. 用户定义完整性

用户定义完整性是针对具体应用语义提出的数据约束。

常见约束：

- 列值非空：`NOT NULL`
- 列值唯一：`UNIQUE`
- 条件检查：`CHECK`

### 5.1 属性上的约束

非空约束：

```sql
Sname CHAR(20) NOT NULL
```

唯一约束：

```sql
SHname VARCHAR(40) UNIQUE
```

检查约束：

```sql
Ssex CHAR(6) CHECK (Ssex IN ('男', '女'))
```

成绩范围：

```sql
Grade SMALLINT CHECK (Grade BETWEEN 0 AND 100)
```

### 5.2 元组上的约束

元组级约束可以涉及多个属性之间的关系。

例子：

```sql
CREATE TABLE Teacher (
    Eno CHAR(8) PRIMARY KEY,
    Ename VARCHAR(20),
    Sal NUMERIC(7,2),
    Deduct NUMERIC(7,2),
    CHECK (Sal + Deduct >= 3000)
);
```

含义：应发工资 `Sal + Deduct` 不低于 3000。

### 5.3 用户定义完整性的检查

当插入元组或修改属性值时，DBMS 会检查用户定义约束是否满足。

若不满足，操作通常被拒绝执行。

## 6. 完整性约束命名子句

SQL 提供 `CONSTRAINT` 子句为完整性约束命名。

格式：

```sql
CONSTRAINT <完整性约束名> <完整性约束>
```

可命名的约束包括：

- `NOT NULL`
- `UNIQUE`
- `PRIMARY KEY`
- `FOREIGN KEY`
- `CHECK`

例子：

```sql
CREATE TABLE Student (
    Sno CHAR(8)
        CONSTRAINT C1 CHECK (Sno BETWEEN '10000000' AND '29999999'),
    Sname CHAR(20)
        CONSTRAINT C2 NOT NULL,
    Sbirthdate DATE
        CONSTRAINT C3 CHECK (Sbirthdate > DATE '1980-01-01'),
    Ssex CHAR(6)
        CONSTRAINT C4 CHECK (Ssex IN ('男', '女')),
    Smajor VARCHAR(40),
    CONSTRAINT StudentKey PRIMARY KEY (Sno)
);
```

优点：

- 便于后续修改和删除约束。
- 便于错误提示和维护。
- 使约束含义更清晰。

## 7. 修改和删除完整性约束

增加约束：

```sql
ALTER TABLE <表名>
ADD CONSTRAINT <约束名> <约束条件>;
```

删除约束：

```sql
ALTER TABLE <表名>
DROP CONSTRAINT <约束名>;
```

例子：

```sql
ALTER TABLE Student
DROP CONSTRAINT C1;
```

```sql
ALTER TABLE Student
ADD CONSTRAINT C1 CHECK (Sno BETWEEN '10000000' AND '29999999');
```

## 8. 断言

### 8.1 断言的含义

断言是数据库级的完整性约束，可以涉及多个表或更复杂的条件。

断言比表约束更全局、更强，但实现和检查成本较高。

### 8.2 定义断言

```sql
CREATE ASSERTION <断言名>
CHECK (<条件>);
```

### 8.3 删除断言

```sql
DROP ASSERTION <断言名>;
```

理解：断言用于表达普通表级 `CHECK` 难以表达的全局约束。

## 9. 触发器

### 9.1 触发器的含义

触发器是由事件触发、自动执行的一段数据库程序。

触发器可以实现更复杂的完整性检查和违约处理。

触发器规则通常表示为：

```text
事件 - 条件 - 动作
```

即：当某个事件发生时，检查条件；若条件成立，则执行动作。

### 9.2 创建触发器

一般格式：

```sql
CREATE TRIGGER <触发器名>
{BEFORE | AFTER} <触发事件> ON <表名>
REFERENCING NEW | OLD AS <变量>
FOR EACH {ROW | STATEMENT}
[WHEN <触发条件>]
<触发动作体>;
```

### 9.3 触发器关键组成

| 组成 | 含义 |
| --- | --- |
| 触发器名 | 标识触发器 |
| 目标表 | 触发器所依附的基本表 |
| 触发事件 | INSERT、DELETE、UPDATE 等 |
| BEFORE / AFTER | 触发时机 |
| FOR EACH ROW | 行级触发器，每影响一行执行一次 |
| FOR EACH STATEMENT | 语句级触发器，一条语句执行一次 |
| WHEN | 触发条件 |
| 触发动作体 | 条件成立时执行的动作 |

### 9.4 NEW 与 OLD

在行级触发器中：

- `NEW` 表示更新或插入后的新值。
- `OLD` 表示更新或删除前的旧值。

注意：语句级触发器通常不能在动作体中使用 `NEW` 或 `OLD` 引用单行值。

### 9.5 触发器的特点

优点：

- 可以实现复杂约束。
- 可以自动处理违约。
- 可以用于审计、安全、业务流程控制等。

风险：

- 触发器动作复杂，不易调试。
- 一个触发器可能激活另一个触发器，形成触发链。
- 设计不当可能造成难以预见的错误。

## 10. 易混概念辨析

### 10.1 实体完整性、参照完整性、用户定义完整性

| 类型 | 约束对象 | 典型 SQL |
| --- | --- | --- |
| 实体完整性 | 主码 | PRIMARY KEY |
| 参照完整性 | 外码 | FOREIGN KEY REFERENCES |
| 用户定义完整性 | 应用语义 | NOT NULL、UNIQUE、CHECK |

### 10.2 NO ACTION 与 CASCADE

| 策略 | 含义 |
| --- | --- |
| NO ACTION | 违反约束时拒绝执行 |
| CASCADE | 对相关表级联执行相应操作 |

### 10.3 CHECK 与 ASSERTION

| 概念 | 作用范围 |
| --- | --- |
| CHECK | 多用于列级、元组级或表级约束 |
| ASSERTION | 数据库级全局约束，可涉及多个表 |

### 10.4 触发器与普通约束

| 概念 | 特点 |
| --- | --- |
| 普通约束 | 声明式，简单清晰 |
| 触发器 | 过程式，功能强但复杂 |
